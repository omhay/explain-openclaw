#### Centralize trusted-proxy control-ui bypass policy ([`0cc3e8137`](https://github.com/openclaw/openclaw/commit/0cc3e8137))

This refactor consolidates the trusted-proxy bypass logic for control-ui connections into a single shared policy, eliminating scattered conditional checks across the gateway auth path. The centralized policy now correctly prevents unpaired node sessions from passing through the control-ui path under trusted-proxy mode. This commit works in conjunction with `ec45c317f` to fully close GHSA-vvgp-4c28-m3jm.

#### Split browser auth hardening paths ([`20c2db210`](https://github.com/openclaw/openclaw/commit/20c2db210))

Browser WebSocket authentication is now split into two distinct code paths: loopback auto-pair (for local connections without shared auth) and full origin-validated auth (for all other browser clients). Previously, both paths shared logic that allowed loopback auto-pair to inadvertently bypass device pairing requirements for browser clients. The split ensures that browser-based connections are subject to proper pairing enforcement.

#### Unify reaction ingress policy guards across channels ([`8f8e46d89`](https://github.com/openclaw/openclaw/commit/8f8e46d89))

Reaction ingress across all channels (Discord, Slack, Signal, Telegram, Mattermost) now applies the same DM policy and allowlist checks that govern inbound messages. Previously, reactions could bypass channel-level access controls because they were processed through a separate, less-guarded code path. The unification closes a class of reaction-based policy bypass across all supported channels.

#### Unify subagent announce delivery pipeline ([`4258a3307`](https://github.com/openclaw/openclaw/commit/4258a3307))

Subagent announcement delivery is consolidated through a single pipeline that handles failures explicitly rather than silently discarding them. Previously, announce delivery failures could leave cleanup state inconsistent, with subagents appearing to be running when they had already completed. The unified pipeline surfaces delivery errors and ensures cleanup state transitions are always triggered.

#### Harden discord and slack reaction ingress authorization ([`aedf62ac7`](https://github.com/openclaw/openclaw/commit/aedf62ac7))

Discord and Slack reaction handlers now enforce RBAC checks identical to those applied to inbound messages. Reactions from senders not in the allowlist or pairing set are rejected, and reactions on channels where DM/group messages are disabled are no longer processed. This directly patches GHSA-354r-7mfh-7rh2 (Discord DM reaction ingress missed dmPolicy/allowFrom checks).

#### Unify reaction auth flow and table-drive tests ([`c6dfa26f0`](https://github.com/openclaw/openclaw/commit/c6dfa26f0))

Signal reaction authorization is refactored to use the same policy evaluation flow as Discord and Slack reactions. The refactor extracts shared auth logic into table-driven tests that verify all combinations of DM policy, group policy, and allowlist configurations. This ensures Signal reactions are consistently gated by the same controls applied to other channels.

#### Clarify denyCommands exact-match guidance ([`42f455739`](https://github.com/openclaw/openclaw/commit/42f455739))

Documentation and tests are updated to clarify that `denyCommands` performs namespace-level exact matching, not shell injection prevention. Operators who configure `denyCommands` expecting it to block `rm -rf /` style payloads will now see documentation and audit warnings that redirect them to proper allowlist-based configurations. This prevents a category of security misconfiguration where operators develop a false sense of protection from shell injection.

#### Prevent silent overflow on parent thread forks ([`eb73e87f1`](https://github.com/openclaw/openclaw/commit/eb73e87f1))

When a Slack DM thread's parent reaches context capacity, the overflow is now surfaced as an explicit error rather than silently dropped. The fix adds detection for `ERR_OVERFLOW` on parent thread forks and creates a new thread branch rather than attempting to append to a full context. This eliminates a class of silent message loss that could leave users unaware that their messages were not delivered (#26905).

#### Preserve agentId through gateway send path ([`2011edc9e`](https://github.com/openclaw/openclaw/commit/2011edc9e))

The `agentId` is now propagated through the entire gateway send pipeline to prevent cross-agent media root confusion. Previously, `agentId` was lost at certain handoff points in the send path, causing the gateway to resolve the wrong agent's media root and potentially serving or writing files to the wrong agent's workspace. This preserves agent isolation guarantees across the gateway.

#### Block agents.files symlink escapes ([`125f4071b`](https://github.com/openclaw/openclaw/commit/125f4071b))

The gateway `agents.files` handler now resolves symlinks using `realpath()` before allowing read or write operations, blocking path traversal via symlinks that point outside the agent's workspace. This directly patches GHSA-fgvx-58p6-gjwc (Gateway agents.files symlink escape allowed out-of-workspace file read/write).

#### Harden browser websocket auth chain ([`c736f11a1`](https://github.com/openclaw/openclaw/commit/c736f11a1))

`attachGatewayWsMessageHandler()` now accepts a `browserRateLimiter` parameter and applies it to connections that present a browser `Origin` header, regardless of whether the client is control-ui or webchat. Loopback addresses are assigned a synthetic non-loopback IP (`198.18.0.1`) for rate-limit accounting, preventing loopback exemptions from bypassing brute-force protection. The `enforceBrowserOriginForAnyClient` flag extends origin validation to all browser-origin connections. This patches GHSA-jmmg-jqc7-5qf4 (HIGH).

#### Require pairing for unpaired operator device auth ([`8d1481cb4`](https://github.com/openclaw/openclaw/commit/8d1481cb4))

The `skipPairingForOperatorSharedAuth` bypass is removed from `attachGatewayWsMessageHandler()`. Previously, operators authenticating via shared token could bypass the device pairing requirement on first connect. The fix restricts pairing skip to `shouldSkipControlUiPairing()` (which covers only the specific control-ui trusted-proxy case). This closes the operator shared-auth pairing bypass documented in GHSA-553v-f69r-656j.

#### Enforce auth before reaction notification enqueue ([`2aa7842ad`](https://github.com/openclaw/openclaw/commit/2aa7842ad))

Signal reaction event handling now calls `resolveDmGroupAccessDecision()` and `resolveEffectiveAllowFromLists()` from `dm-policy-shared.js` before enqueuing reaction notifications. Previously, access checks were deferred until after the reaction was enqueued, creating a window where unauthorized reactions could trigger processing. The fix ensures `dmAllowed` is computed and checked before any reaction notification is created. This patches GHSA-792q-qw95-f446.

#### Harden browser trace/download temp path handling ([`496a76c03`](https://github.com/openclaw/openclaw/commit/496a76c03))

A new `resolveWritablePathWithinRoot()` function in `src/browser/paths.ts` validates the full path chain — root directory, parent directory, and target file — using `fs.lstat()` and `fs.realpath()` to reject any symlinks in the path. The root directory itself is also checked to ensure it is not a symlink. This prevents an attacker from placing a symlink in the browser trace/download temp directory to redirect writes to arbitrary host paths. Patches GHSA-36h3-7c54-j27r.

#### Enforce telegram reaction authorization ([`e56b0cf1a`](https://github.com/openclaw/openclaw/commit/e56b0cf1a))

The commit introduces `authorizeTelegramEventSender()` with table-driven authorization rules (`TELEGRAM_EVENT_AUTH_RULES`) that evaluate DM policy and group allowlist membership before allowing Telegram reaction events to be processed. Both DM and group access decisions are resolved via `resolveTelegramEventAuthorizationContext()` before the reaction handler executes, and the sender is denied early if not permitted. This closes GHSA-qj22-xqjr-v83v (Telegram message_reaction authorization bypass).

#### Harden nextcloud-talk webhook replay handling ([`d512163d6`](https://github.com/openclaw/openclaw/commit/d512163d6))

A new `createNextcloudTalkReplayGuard()` function in `extensions/nextcloud-talk/src/replay-guard.ts` tracks recently processed webhook payloads using normalized origin and nonce values. Duplicate webhook deliveries are detected and rejected before reaching inbound processing, preventing Nextcloud Talk webhook replay from triggering duplicate agent invocations. Patches GHSA-r9q5-c7qc-p26w.

#### Gate interactive system events by sender auth ([`ce8c67c31`](https://github.com/openclaw/openclaw/commit/ce8c67c31))

Slack modal and button interaction handlers now run sender authorization through `src/slack/monitor/auth.ts` before processing the callback payload. Previously, interactive callbacks (block actions, modal submissions) bypassed the configured sender checks applied to regular Slack messages. The fix adds `modal-metadata.ts` helpers to associate callback IDs with the originating conversation and sender, enabling per-interaction auth decisions. Patches GHSA-x2ff-j5c2-ggpr (HIGH).

#### Block trusted-proxy control-ui node bypass ([`ec45c317f`](https://github.com/openclaw/openclaw/commit/ec45c317f))

`src/gateway/server/ws-connection/message-handler.ts` is updated to block trusted-proxy bypass for node (non-browser) client connections to the control-ui endpoint. Previously, node clients could exploit the trusted-proxy code path to connect to control-ui without completing device pairing. Together with `0cc3e8137`, this fully closes the bypass documented in GHSA-vvgp-4c28-m3jm.

#### Bind file consent invokes to conversation ([`347f7b955`](https://github.com/openclaw/openclaw/commit/347f7b955))

MS Teams file consent accept/decline handlers in `extensions/msteams/src/monitor-handler.ts` now bind to the specific conversation ID recorded when the file upload was initiated. Previously, pending file consent handlers could be triggered by any MS Teams interaction in any conversation, allowing cross-conversation consumption of pending file uploads. This patches GHSA-j26j-7qc4-3mrf.

#### Revalidate upload paths at use time ([`ef326f5cd`](https://github.com/openclaw/openclaw/commit/ef326f5cd))

Browser upload paths are now resolved via `fs.realpath()` at the point of actual file use, not just at the point of validation. This prevents TOCTOU attacks where a path passes the initial boundary check but is swapped via hardlink or symlink before the file operation executes. The canonical path is stored and used for all subsequent operations within the same upload flow. Patches GHSA-3jx4-q2m7-r496 (HIGH).
