### Key Security Changes

1. **`ffa63173e`** — **Subsystem Logger Migration for Security Event Logging:**

   Migrated `console.warn`/`console.error`/`console.info` calls to the structured `createSubsystemLogger` API across 16 production source files in `src/agents/`. The security-significant change is that `[Security]`-prefixed warnings in two paths now emit through `log.warn(...)` instead of raw `console.warn`:

   - `src/agents/sandbox/docker.ts:259` — `buildSandboxCreateArgs()`: env-var blocking notifications now at lines 297–300 via `log.warn("Blocked sensitive environment variables: ...")` and `log.warn("Suspicious environment variables: ...")`
   - `src/agents/skills/env-overrides.ts:77` — `applySkillConfigEnvOverrides()`: blocked skill env-override notifications now at lines 120–123 via `log.warn("Blocked skill env overrides: ...")`

   Routing security audit events through the structured subsystem logger improves log consistency and auditability for operators monitoring environment variable blocking.

   **Relevance to audit claims:** No direct match to the 16 audit claims; defense-in-depth improvement to the audit trail for the env-var blocklist (Legitimate Gap 1 area — gateway-side blocklist already closed; this hardens the log path for both Docker sandbox and skill override enforcement).

2. **`cc2ff6894`** — **Test Coverage Hardening for Device Auth and SSRF:**

   Refactored 24 test files to use `beforeAll` module imports. The security-relevant changes:

   - **Device auth integration tests strengthened** (`src/gateway/server.talk-config.e2e.test.ts`): Added `createFreshOperatorDevice()` helper (line 7) that exercises the full device-auth signing pipeline — `buildDeviceAuthPayload` (`src/gateway/device-auth.ts:13`), `loadOrCreateDeviceIdentity` (`src/infra/device-identity.ts:65`), and `signDevicePayload` (`src/infra/device-identity.ts:125`) — injecting real signed device credentials into gateway connection tests. Strengthens coverage of scope enforcement for the `operator.talk.secrets` scope gate.

   - **SSRF classification tests consolidated** (`src/infra/net/ssrf.test.ts`): Merged `it.each` tests for `isPrivateIpAddress` (`src/infra/net/ssrf.ts:308`) into grouped loops covering private IPs, public IPs, malformed IPv6, legacy IPv4, and non-IP hostnames. Functionally equivalent coverage.

   - **Attachment validation test coverage reduced** (`src/gateway/chat-attachments.test.ts`): Removed test cases for invalid base64 content rejection and max-byte size limit enforcement in `buildMessageWithAttachments`. The production validation at `src/gateway/chat-attachments.ts:80` (`isValidBase64` check and `estimateBase64DecodedBytes` size gate) is unchanged; negative test paths are no longer exercised.

   **Relevance to audit claims:** Audit 2 Claim 4 (DNS rebinding SSRF via web-fetch) — SSRF test consolidation maintains coverage of `isPrivateIpAddress` guard that protects against internal IP literal targeting.

3. **`b34097f62`** — **MSTeams Redirect Open-Redirect / SSRF Fix:**

   Adds explicit redirect-URL allowlist enforcement for MSTeams SharePoint media downloads in `extensions/msteams/src/attachments/graph.ts`. Two new functions parse the HTTP Location header before any redirect is followed:

   - `isRedirectStatus` (line 35): identifies 3xx redirect responses
   - `readRedirectUrl` (line 39): extracts and validates the Location header

   `downloadMSTeamsGraphMedia` (line 235) now calls these before following any redirect; if the resolved redirect target is not in the configured `allowHosts` list, the download is aborted. A pre-flight `isUrlAllowed` check (line 287) also blocks the initial content URL before any request is issued.

   This closes an open-redirect / SSRF vector where a malicious SharePoint URL could cause the gateway to follow HTTP redirects to arbitrary external hosts. **Addresses Audit 2 Claim 4** (DNS rebinding SSRF via web-fetch — MSTeams media download path).

4. **`7c500ff62`** — **Control-UI Static Path Hardening — TOCTOU / Symlink Fix:**

   Replaces the naive `fs.existsSync` / `fs.statSync` file-serve path in `src/gateway/control-ui.ts` with a defence-in-depth sequence:

   - `isContainedPath` (line 204): verifies request path is contained within the configured root
   - `isExpectedSafePathError` (line 209): classifies safe vs. unexpected open errors
   - `areSameFileIdentity` (line 215): compares pre-open and post-open inode identity to defeat TOCTOU symlink-swap attacks
   - `resolveSafeControlUiFile` (line 219): resolves both root and target to their real paths via `fs.realpathSync`, enforces containment, opens with `O_NOFOLLOW` to refuse symlinks, and verifies inode identity
   - `handleControlUiHttpRequest` (line 273): both the direct file-serve path and the SPA `index.html` fallback now route through `resolveSafeControlUiFile`

   Closes a path-traversal and symlink-attack vector on the local Control UI HTTP server.

5. **`90a378ca3`** — **macOS Exec Allowlist: Double-Quoted Shell Substitution Bypass Fix:**

   Extends the shell-injection guard in `apps/macos/Sources/OpenClaw/ExecCommandResolution.swift` to cover command and process substitution characters (backtick, `$(...)`) inside double-quoted strings:

   - `shouldFailClosedForShell` (line 244): accepts an `inDouble` context flag; the call site now fires whenever `!inSingle` rather than only when `!inSingle && !inDouble`

   Previously, an allowlist pattern of the form `"$(cmd)"` would have passed the guard, potentially enabling shell injection through the exec allowlist. **Addresses Audit 2 Claim 7** (shell injection via incomplete regex in exec allowlist).

6. **`dd41fadca`** — **macOS Exec Allowlist: Enforce Path-Only Patterns:**

   Enforces that all exec allowlist entries must be path patterns (containing `/`, `~`, or `\`) at two points in `apps/macos/Sources/OpenClaw/ExecApprovals.swift`:

   - `addAllowlistEntry` (Swift:439): returns early if `isPathPattern` check fails
   - `normalizeAllowlistEntries` (line 617): filters non-path entries during `updateAllowlist` (line 490)
   - `migrateLegacyPattern` (line 584): upgrades existing bare-name entries to their resolved path if available

   In `ExecAllowlistMatcher.swift`, the fallback branch that matched patterns against the bare `executableName` (without a path component) is removed entirely — only path-anchored patterns can match. This prevents bypass of the exec allowlist via bare command-name entries. **Partially addresses Audit 2 Claim 7**.

7. **`2028ca442`** — **macOS Exec Allowlist: Unified Validation Pipeline:**

   Introduces a structured validation type hierarchy in `apps/macos/Sources/OpenClaw/ExecApprovals.swift` and centralises all allowlist pattern validation through a single enforcer:

   - `ExecAllowlistPatternValidationReason` enum (line 93): reasons for rejection
   - `ExecAllowlistPatternValidation` enum (line 107): `valid` / `invalid(reason)` cases
   - `ExecAllowlistRejectedEntry` struct (line 112): rejected entry with reason
   - `validateAllowlistPattern` (line 694): central validator — trims pattern and enforces `containsPathComponent` (line 732)

   The 245-line refactor routes every read, write, and merge path — `normalizeAllowlistEntries`, `migrateLegacyPattern`, `addAllowlistEntry`, `mergeAgents`, `normalizedPattern` — through the new centralized validator, eliminating inconsistency between the multiple prior ad-hoc `isPathPattern` checks. This is the capstone of a three-commit series (with `90a378ca3` and `dd41fadca`). **Addresses Audit 2 Claim 7**.

8. **`73d93dee6`** — **Inbound Media Max-Bytes Enforcement During Remote Fetch:**

   Migrates attachment download paths across five extensions (BlueBubbles, MSTeams download, MSTeams graph, Discord, Telegram) to route through the shared `channel.media.fetchRemoteMedia` API, which enforces `maxBytes` during streaming rather than after full buffering:

   - `downloadBlueBubblesAttachment` (`extensions/bluebubbles/src/attachments.ts:88`): previously buffered the full response before comparing `buf.byteLength > maxBytes`; now delegates to `fetchRemoteMedia` (`:90`), enforcing the limit during fetch
   - `fetchWithAuthFallback` (`extensions/msteams/src/attachments/download.ts:253`): similarly delegates size enforcement to the runtime

   This closes a DoS vector where a malicious server could exhaust gateway memory by returning an oversized media attachment.
