#### Gateway WS Unauthorized Flood Guard ([`7fb69b7cd`](https://github.com/openclaw/openclaw/commit/7fb69b7cd))

The gateway WebSocket connection handler now tracks unauthorized request counts per connection via the new `UnauthorizedFloodGuard` class (`src/gateway/server/ws-connection/unauthorized-flood-guard.ts`). The guard's `registerUnauthorized()` method increments a per-connection counter and returns a decision object with `shouldClose` and `shouldLog` flags. Connections exceeding `DEFAULT_CLOSE_AFTER` (10) unauthorized requests are forcibly closed to terminate floods. Built-in log suppression (`DEFAULT_LOG_EVERY = 100`) logs only the first request and every 100th to avoid noise. The `UnauthorizedFloodGuard` instance is created per connection inside `message-handler.ts` and consulted for each unauthorized message event. Prior to this commit, a client could send unlimited unauthorized WS messages without consequence, creating a low-cost DoS vector for connections that pass the handshake but lack valid auth.

#### Hide Direct-Chat Metadata Without Sender-ID Sentinel ([`2247b8121`](https://github.com/openclaw/openclaw/commit/2247b8121))

The `buildInboundMeta()` function in `src/auto-reply/reply/inbound-meta.ts` now tests `isDirect` (true when `chatType` is `"direct"` or absent) before populating chat-group fields. For direct/private chats, `message_id`, `message_id_full`, `reply_to_id`, `sender_id`, `sender`, and `conversation_label` are all set to `undefined`, omitting them entirely from structured inbound metadata. Previously, direct-chat metadata fields including `sender_id` were leaking into structured metadata for auto-reply processing, creating unnecessary information exposure when metadata is forwarded or logged.

#### Config Write Immutability for unsetPaths ([`f208518cb`](https://github.com/openclaw/openclaw/commit/f208518cb))

The write path in `src/config/io.ts` was refactored to prevent mutation of caller-owned config objects. The new `unsetPathForWriteAt()` helper traverses the config tree immutably: at each level, it creates new arrays (`.slice()` + `.splice()`) or new objects (`{...spread}` + `delete`) rather than modifying in place. The `WRITE_PRUNED_OBJECT` sentinel symbol marks nodes to be removed at the parent level after recursive processing. The `hasOwnObjectKey()` guard uses `Object.prototype.hasOwnProperty.call()` to block prototype-chain traversal. Previously, `unsetPaths` operations could have silently mutated the caller's config object as a side effect of write processing, creating unexpected behavioral changes for security-sensitive config paths managed by the gateway.

#### Obfuscated Command Detection ([`0e28e50b4`](https://github.com/openclaw/openclaw/commit/0e28e50b4))

A new module `src/infra/exec-obfuscation-detect.ts` exports `detectCommandObfuscation()`, which tests commands against `OBFUSCATION_PATTERNS` — a table-driven list detecting base64-piped-to-shell patterns (e.g., `echo <payload> | base64 -d | sh`), `eval`-based execution, `source`/`.`-piped file loading, and similar obfuscation techniques. The function returns an `ObfuscationDetection` object with `detected` and `reasons` fields. Both `src/agents/bash-tools.exec-host-gateway.ts` and `src/agents/bash-tools.exec-host-node.ts` now call `detectCommandObfuscation()` before deciding whether to require approval. If obfuscation is detected, the command is treated as requiring explicit approval and receives deny reason `"approval-timeout (obfuscation-detected)"` on timeout. This directly strengthens the allowlist (Audit 2 Claim 7): previously, an attacker who could run commands could embed base64-encoded payloads to bypass allowlist checks since the outer command appeared benign.

#### Pre-Commit and CI Security Enforcement ([`1be889733`](https://github.com/openclaw/openclaw/commit/1be889733))

`.pre-commit-config.yaml` and `.github/workflows/ci.yml` gained three security audit hooks. (1) `detect-private-key` runs `detect-secrets==1.5.0` to scan all committed files for private keys and API tokens; the hook excludes `.secrets.baseline`, test files, and Fastfile to reduce false positives. (2) A `zizmor` GitHub workflow auditor checks changed `.github/workflows/*.yml` files for security issues; it conditionally skips when no workflow files changed, preventing unnecessary failures on non-workflow PRs. (3) `pnpm-audit-prod` runs `pnpm audit --prod --audit-level=high` against production dependencies to surface high-severity CVEs. These controls enforce credential hygiene and dependency security at both commit-time (pre-commit hooks) and CI time (GitHub Actions job), providing defense-in-depth against accidental secret commits and vulnerable dependencies.

#### Skill Packaging Path Traversal Guard ([`844924cf8`](https://github.com/openclaw/openclaw/commit/844924cf8))

`skills/skill-creator/scripts/package_skill.py` added two path-hardening guards during skill archive creation. First, symlinks are now skipped: `if file_path.is_symlink(): continue` prevents symlinks pointing outside the skill directory from pulling in external files. Second, a new `_is_within(path, root)` helper validates each file's resolved path using `path.relative_to(root)` (raises `ValueError` for escapes); if a resolved path falls outside the skill root, packaging is aborted with an error. Without these guards, a crafted skill directory could use path traversal or symlink chaining to include arbitrary files from the host filesystem in the packaged skill archive, touching on path traversal patterns described in Audit 1 Claim 6.

#### HTML Gallery Stored XSS Prevention ([`f3adf142c`](https://github.com/openclaw/openclaw/commit/f3adf142c))

`skills/openai-image-gen/scripts/gen.py` generates an HTML gallery page for AI-generated images, writing user-supplied prompt text and file paths directly into HTML. The fix imports `html_escape` from Python's `html` standard library and applies it at all injection points: `html_escape(it["file"], quote=True)` on image file paths in `src` and `href` attributes (with `quote=True` to escape double quotes, preventing attribute injection), `html_escape(it["prompt"])` in `figcaption` text, and `html_escape(out_dir.as_posix())` in the output path display. Without this fix, an attacker controlling an image prompt or file name could inject malicious HTML or JavaScript into the gallery page — a stored XSS vulnerability persisting across gallery renders.

#### OTEL Log Export Sensitive Data Redaction — CWE-532 ([`7fab4d128`](https://github.com/openclaw/openclaw/commit/7fab4d128))

The diagnostics-otel extension (`extensions/diagnostics-otel/src/service.ts`) ships log, span, and event data to external OTLP collectors. Before this fix, all log messages and span attributes were sent verbatim, leaking API keys, session tokens, and other sensitive strings to third-party observability platforms (CWE-532). The fix exports `redactSensitiveText` from `openclaw/plugin-sdk` and applies it at all OTEL export boundaries: log message `body`, all string-typed span attributes via the new `redactOtelAttributes()` helper (numeric and boolean values pass through unchanged), error message strings, and reason fields on diagnostic events. `redactSensitiveText` recognizes common provider token patterns (`sk-*`, `ghp_*`, `gsk_*`, `AIza*`, and others). This strengthens Audit 1 Claim 1 controls (credential redaction) by extending coverage to the OTEL observability export path.

#### Telegram Delivery Target Format Validation ([`78f801e24`](https://github.com/openclaw/openclaw/commit/78f801e24))

The cron job executor in `src/cron/service/jobs.ts` gained input validation for Telegram delivery targets before job execution. A new `validateTelegramDeliveryTarget()` function applies two regex checks: `TELEGRAM_TME_URL_REGEX` rejects `t.me/` URLs (chat IDs should be used, not web URLs), and `TELEGRAM_SLASH_TOPIC_REGEX` rejects slash-delimited topic formats like `-1001234567890/123` (the correct delimiter is colon, not slash). Invalid targets throw a descriptive error listing valid formats. Without validation, misformatted targets would fail silently or produce confusing delivery errors at runtime; early validation surfaces misconfiguration at job creation/scheduling time.
