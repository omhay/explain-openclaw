#### Harden sandbox fs dash-path regression coverage (#25891) (thanks @albertlieyingadrian) ([`c7ae4ed04`](https://github.com/openclaw/openclaw/commit/c7ae4ed04))

This commit is a follow-up to [`5e3502df5`](https://github.com/openclaw/openclaw/commit/5e3502df5). Rather than prepending `./` via the ensurePathNotInterpretedAsOption helper (removed) introduced in that prior commit, this commit replaces the approach by relying on the shell's `--` (end-of-options) double-dash delimiter to prevent leading-hyphen paths from being misinterpreted as shell options in sandbox filesystem operations. The helper function is removed from `src/agents/sandbox/fs-bridge.ts`, and regression tests are added to `fs-bridge.test.ts` verifying that paths like `--leading.txt` are correctly resolved to `/workspace/--leading.txt` within the sandbox without triggering `Syntax error: ; unexpected` failures. This simplification trusts the shell's well-established `--` convention rather than rewriting path strings.

#### Prevent shell option interpretation for paths with leading hyphens ([`5e3502df5`](https://github.com/openclaw/openclaw/commit/5e3502df5))

Sandbox filesystem operations on auto-generated inbound media filenames (which contain the `---` pattern, e.g. `file_1095---f00a04a2-99a0-4d98-99b0-dfe61c5a4198.ogg`) were failing with `Syntax error: ; unexpected` because the `sh` shell interpreted the leading `-` as an option flag. This commit adds the ensurePathNotInterpretedAsOption helper to `src/agents/sandbox/fs-bridge.ts`, prepending `./` to any path that begins with `-`, applied across `readFile`, `writeFile`, `mkdirp`, `remove`, and `move` operations. The follow-on commit `c7ae4ed04` later replaced this path-rewriting approach with shell `--` double-dash, but the two commits together produced comprehensive regression test coverage for dash-prefixed sandbox paths.

#### Harden voice DAVE receive reliability ([`9cd50c51b`](https://github.com/openclaw/openclaw/commit/9cd50c51b))

Hardens Discord's DAVE (End-to-End Encryption) decryption reliability with three new constants in `src/discord/voice/manager.ts`: `DECRYPT_FAILURE_WINDOW_MS` (30,000 ms sliding window), `DECRYPT_FAILURE_RECONNECT_THRESHOLD` (3 failures within the window), and `DECRYPT_FAILURE_PATTERN` (regex matching `DecryptionFailed(` error strings). `VoiceSessionEntry` gains three tracking fields — `decryptFailureCount`, `lastDecryptFailureAt`, and `decryptRecoveryInFlight` — enabling windowed failure rate detection. When the threshold is exceeded within the window, the voice session reconnects automatically, preventing persistent decryption failures from silently degrading voice quality. New config fields `daveEncryption` (boolean) and `decryptionFailureTolerance` (number) allow gateway operators to tune DAVE encryption behavior and recovery tolerance per configuration. Comprehensive regression tests validate stale session guards and voice listener cleanup.

#### Prevent cross-channel reply routing in shared sessions ([`455fbc6b6`](https://github.com/openclaw/openclaw/commit/455fbc6b6))

Introduces `turnSourceChannel` (along with paired `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId`) to both `src/infra/outbound/targets.ts` and `src/infra/outbound/agent-delivery.ts` to record which channel originated a given conversation turn. `resolveSessionDeliveryTarget()` now prefers the turn-originating channel over the session's last-active channel when resolving outbound routing. Without this fix, a session shared across multiple channels (e.g. WhatsApp and Telegram) could route replies to the wrong channel depending on timing — effectively allowing cross-channel reply spoofing. Tests in `targets.test.ts` cover the three cases: `turnSourceChannel` takes precedence over session `lastChannel`, fallback to `lastChannel` when `turnSourceChannel` is absent, and explicit `requestedChannel` overrides both. This addresses the session-level channel isolation concern tracked in GH#24152 (Audit 2 Claim 5).

#### Suppress reasoning payload leakage in WhatsApp replies ([`039713c3e`](https://github.com/openclaw/openclaw/commit/039713c3e))

Adds `shouldSuppressReasoningReply()` to `src/web/auto-reply/deliver-reply.ts` to block Gemini 3.1 reasoning/thinking traces from reaching WhatsApp users. The guard applies two checks: whether the `isReasoning` flag is set on the `ReplyPayload`, and whether the text begins with the `REASONING_PREFIX` constant (`"reasoning:"`, case-insensitive after trimming leading whitespace). If either check matches, `deliverWebReply()` returns early without invoking `msg.reply()` or `msg.sendMedia()`, and logs a debug line via `whatsappOutboundLog`. This guard fires before any message chunking or delivery logic, ensuring no partial reasoning content leaks. Two tests cover the `isReasoning: true` path and the prefix-text detection path.

#### Block env depth-overflow approval bypass ([`57c9a1818`](https://github.com/openclaw/openclaw/commit/57c9a1818))

Patches `resolveDispatchWrapperExecutionPlan()` in `src/infra/exec-wrapper-resolution.ts` to enforce the `MAX_DISPATCH_WRAPPER_DEPTH` cap (4) after the wrapper unwrapping loop. When `wrappers.length >= maxDepth` and the remaining invocation still resolves to `"blocked"` or `"unwrapped"`, the function returns `policyBlocked: true` with the offending wrapper recorded in `blockedWrapper`. Before this fix, a chain of the form `/usr/bin/env env env env env <blocked-cmd>` could exhaust the depth cap without triggering a block, allowing the inner command to execute as though it were not wrapped in a dispatch wrapper. The fix ensures that depth-cap overflow itself triggers policy enforcement, preventing approval gating from being bypassed via recursive `env` invocation. This closes the shell-wrapper depth-overflow vector described in Audit 2 Claim 7.

#### Align windows safe-open file identity checks ([`943b8f171`](https://github.com/openclaw/openclaw/commit/943b8f171))

Extracts file identity verification logic into a new dedicated module `src/infra/file-identity.ts`, centralizing the `sameFileIdentity()` function used by safe-open operations. The function accepts two `Stats` objects and an optional platform string; on Windows it skips the unreliable `dev` (device ID) field and checks only `ino` (inode), while on Unix it requires both to match. Both `src/infra/fs-safe.ts` and `src/infra/safe-open-sync.ts` are updated to import from the new module. The new test file `src/infra/file-identity.test.ts` covers Linux full-match, inode-mismatch, device-mismatch, Windows `dev=0` bypass, and bigint variant cases. This refactor makes file identity checking consistent and testable across platforms, strengthening the safe-open controls relevant to Audit 1 Claims 5 and 6.

#### Skip unreliable dev comparison in fs-safe openVerifiedLocalFile ([`7455ceecf`](https://github.com/openclaw/openclaw/commit/7455ceecf))

On Windows, `handle.stat()` and `fs.lstat()` can return different `dev` values for the same file, causing false-positive `path-mismatch` errors in `openVerifiedLocalFile()` during local media file access. This commit introduces an inline statsMatch helper (later removed) in `src/infra/fs-safe.ts` that always compares `ino` (inode) values but skips the `dev` comparison when `process.platform === 'win32'`. Full `ino + dev` comparison is maintained on Unix. This fixes the Windows-specific safe-open failures reported in GH#25699. The follow-on commit `943b8f171` later extracted and generalized this platform-aware identity check into `src/infra/file-identity.ts` as `sameFileIdentity()`, superseding the inline helper (Audit 1 Claim 5).

#### Harden iMessage echo dedupe and reasoning suppression ([`2a11c09a8`](https://github.com/openclaw/openclaw/commit/2a11c09a8))

Extends iMessage echo deduplication to use message IDs in addition to text matching, and adds reasoning suppression at the outbound payload normalization layer. In `src/imessage/monitor/inbound-processing.ts`, `resolveIMessageInboundDecision()` now checks both `messageText` and the inbound `message.id` field via the updated `echoCache.has()` interface, and `describeIMessageEchoDropLog()` includes the message ID in its log output. In `src/imessage/monitor/deliver.ts`, `deliverReplies()` records the returned `messageId` from `sendMessageIMessage()` in the echo cache alongside the text, enabling ID-based deduplication for subsequent echoes. In `src/infra/outbound/payloads.ts`, `normalizeReplyPayloadsForDelivery()` now filters out payloads with `isReasoning: true` before delivery, preventing reasoning traces from leaking through the iMessage outbound pipeline — complementing the same guard added to WhatsApp in `039713c3e`.

#### Default target none and internalize relay prompts ([`e2362d352`](https://github.com/openclaw/openclaw/commit/e2362d352))

Changes `resolveHeartbeatDeliveryTarget()` in `src/infra/outbound/targets.ts` to initialize the local `target` variable as `"none"` instead of `"last"`, and updates `DEFAULT_HEARTBEAT_TARGET` in `src/infra/heartbeat-runner.ts` to `"none"`. This makes heartbeat delivery conservative by default — the gateway will not implicitly pick up the last-active channel when no explicit heartbeat target is configured. Documentation in `docs/automation/cron-vs-heartbeat.md`, `docs/gateway/configuration-reference.md`, and `docs/gateway/heartbeat.md` is updated to reflect the new default. This is a defense-in-depth change ensuring heartbeat messages are not silently sent to unexpected channels in fresh or minimal configurations. Related tests in `heartbeat-runner.returns-default-unset.test.ts` confirm the `"none"` default behavior.
