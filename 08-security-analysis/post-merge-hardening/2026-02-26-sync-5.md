### SSRF Dispatcher Family Policy Fix ([`30fd2bbe1`](https://github.com/openclaw/openclaw/commit/30fd2bbe1))

Removes `autoSelectFamily: true` and `autoSelectFamilyAttemptTimeout: 300` from `createPinnedDispatcher()`. Previously, the pinned dispatcher overrode the global IPv4/IPv6 family selection policy with forced automatic family selection, bypassing the SSRF guard's intent of restricting which network families are used for outbound connections. This fix ensures the pinned dispatcher inherits the global family policy configured at the Node.js Agent level rather than overriding it.

**Audit 2 Claim 4 relation:** Strengthens the DNS rebinding/SSRF mitigation by ensuring pinned hostname dispatchers cannot bypass the globally-configured network family policy.

### Silent Reply Token Matching Tightened ([`2f2110a32`](https://github.com/openclaw/openclaw/commit/2f2110a32) + [`e64d72299`](https://github.com/openclaw/openclaw/commit/e64d72299))

`isSilentReplyText()` previously used prefix and suffix regex patterns (`^\\s*${escaped}(?=$|\\W)` and `\\b${escaped}\\b\\W*$`) that would incorrectly suppress substantive replies when a model appended `SILENT_REPLY_TOKEN` ("NO_REPLY") to legitimate content. The fix replaces these with a single whole-string pattern (`^\\s*${escaped}\\s*$`), ensuring only responses consisting entirely of the token are suppressed.

A companion commit introduces `isSilentReplyPrefixText()` to handle prefix-only token detection consistently across the streaming pipeline (via `streaming-directives.ts`) and final reply validation (via `agent-runner-execution.ts`). `HEARTBEAT_TOKEN` handling is updated to use the same prefix-text logic, ensuring consistent suppression behavior across all token types.

**Security relevance:** Silent reply suppression logic gates whether a model's response is delivered to users. Incorrect suppression could cause legitimate responses to be silently dropped, while incorrect non-suppression could cause `NO_REPLY` tokens to appear in user-visible output.

### Doctor: Account-Scoped allowFrom Verification ([`1ffc31983`](https://github.com/openclaw/openclaw/commit/1ffc31983))

Adds comprehensive test coverage verifying that `allowFrom` authorization lists in Discord, Slack, and Telegram multi-account configurations are correctly scoped. Tests confirm that account-level `allowFrom` (e.g., `channels.discord.accounts.default.allowFrom`) overrides top-level lists and that named accounts do not inherit the default account's allowlist, preventing accidental privilege escalation in multi-bot deployments. The commit also renames `normalizeLegacyConfigValues` to `normalizeCompatibilityConfigValues` throughout the doctor flow (terminology update).

**Security relevance:** `allowFrom` is the primary access control mechanism for channel-level message authorization. This commit verifies the scoping semantics are correct rather than fixing a regression.

### Doctor Integrity: Slash Session Filtering ([`a690b6239`](https://github.com/openclaw/openclaw/commit/a690b6239))

Adds `isSlashRoutingSessionKey()` using `parseAgentSessionKey()` and a regex pattern (`/^[^:]+:slash:[^:]+(?:$|:)/`) to filter slash-routed sessions from the recent-sessions transcript integrity check in `noteStateIntegrity()`. Previously, slash routing sessions — which legitimately lack persistent transcripts — were included in the integrity count, producing false-positive warnings. The integrity report now excludes these sessions from the denominator.

**Security relevance:** Integrity check accuracy; prevents false warnings from masking real transcript integrity issues.
