#### Cache bootstrap snapshots per session key ([`40db3fef4`](https://github.com/openclaw/openclaw/commit/40db3fef4))

Adds per-session isolation for workspace bootstrap file caching (AGENTS.md, SOUL.md, etc.). A new `bootstrap-cache.ts` module introduces `getOrLoadBootstrapFiles()`, which stores loaded bootstrap files in a `Map` keyed by `sessionKey`, replacing direct `loadWorkspaceBootstrapFiles()` calls in `bootstrap-files.ts`. A companion `clearBootstrapSnapshot()` function purges the cached snapshot for a given session; it is wired into `ensureSessionRuntimeCleanup()` so cached bootstrap state is evicted when a session ends.

Without this change, concurrent sessions sharing the same process could in principle read stale bootstrap files from a sibling session if the underlying loader had any process-level caching. The new Map-per-session model prevents cross-session context leakage of workspace policy files.

#### Use gateway health status for memory search key check ([`8d6925147`](https://github.com/openclaw/openclaw/commit/8d6925147))

Adds a gateway-side embedding readiness probe so the doctor command can check memory search health without requiring the CLI process to hold API keys. The new gateway method `"doctor.memory.status"` is handled by the gateway doctor handler and called from the CLI via `probeGatewayMemoryStatus()`. The result is passed to `noteMemorySearchHealth()`, which adjusts diagnostic messaging based on whether the gateway confirms the embedding provider is reachable.

The method is added to `READ_SCOPE` in `method-scopes.ts` and `server-methods-list.ts`, ensuring it is accessible only to authenticated read-scope clients and not via unauthenticated paths. This reinforces the gateway method authorization model while improving observability.

#### Derive relay auth token from gateway token in Chrome extension ([`216d99e58`](https://github.com/openclaw/openclaw/commit/216d99e58))

Changes the Chrome extension relay handshake from forwarding the raw gateway token to computing an HMAC-SHA256 derived token. The extension background script gains a `deriveRelayToken()` function (in `chrome-extension-background-utils`) that signs the message `"openclaw-extension-relay-v1:<port>"` using the gateway token as the key, producing a port-scoped derived credential. This prevents the raw gateway token from being sent over the relay WebSocket connection.

CORS-sensitive validation requests from the extension options page are now delegated to the background service worker, eliminating OPTIONS preflight 401 failures that were occurring when the options page tried to validate the relay directly.

This commit (client-side derivation) is paired with [`bb8f538cd`](#browser-relay-accept-raw-gateway-token-in-extension-auth-bb8f538cd) (server-side acceptance logic). Together they implement the full HMAC relay token protocol.

#### Include platform and reason in node command rejection error ([`d00d814ad`](https://github.com/openclaw/openclaw/commit/d00d814ad))

Improves diagnostic clarity for node command rejections in the gateway. A new helper `buildNodeCommandRejectionHint()` in `server-methods/nodes.ts` constructs structured error messages that include the node platform and the specific rejection reason — distinguishing between "command not declared by node," "not in platform allowlist," and "capability not advertised." The hint is used in both the `browser.proxy` and `nodes` handlers.

This is a diagnostic improvement; it does not change allowlist logic or authorization checks. The additional context helps operators distinguish configuration errors from active policy denials without exposing the allowlist contents themselves.

#### Add kimi web_search provider ([`e02c470d5`](https://github.com/openclaw/openclaw/commit/e02c470d5))

The new provider introduces an outbound HTTP surface to Kimi's API endpoint. API keys are resolved through the same configuration path as other providers; no keys are hardcoded or stored beyond the session config.

#### Browser relay: accept raw gateway token in extension auth ([`bb8f538cd`](https://github.com/openclaw/openclaw/commit/bb8f538cd))

Implements the server-side half of the HMAC relay token protocol introduced in [`216d99e58`](#derive-relay-auth-token-from-gateway-token-in-chrome-extension-216d99e58). The new `resolveRelayAcceptedTokensForPort()` function in `extension-relay-auth.ts` computes the HMAC-SHA256 derived relay token using `deriveRelayAuthToken(gatewayToken, port)` with message `"openclaw-extension-relay-v1:<port>"` and returns both the derived token and, for backward compatibility, the raw gateway token (when the two differ). The relay server in `extension-relay.ts` builds a `relayAuthTokens` Set from this list and checks each incoming WebSocket auth token via `getRelayAuthTokenFromRequest()` against the set.

This design provides a grace period: existing extensions using the raw gateway token continue to authenticate while updated extensions send the narrower port-scoped derived credential. The derived token prevents a relay credential from being replayed against other ports or gateway endpoints.
