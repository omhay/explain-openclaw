#### Unify path alias guard policies ([`de61e9c97`](https://github.com/openclaw/openclaw/commit/de61e9c97))

Path alias guards determine whether symlinks and hardlinks in file paths are permitted as path aliases during sandbox and workspace operations. Before this commit, the guard parameters (`allowFinalSymlink`, `allowFinalHardlink`) were scattered across agents, sandbox, and patch modules as separate boolean arguments. This refactor extracts the `PATH_ALIAS_POLICIES` constant and `assertNoPathAliasEscape` function into a new centralized `src/infra/path-alias-guards.ts` module, replacing the scattered boolean parameters with the `PathAliasPolicy` type across all callers. Centralization ensures consistent enforcement of path alias rules across every code path that handles untrusted file system operations.

#### Unify exec approval request matching ([`92eb3dfc9`](https://github.com/openclaw/openclaw/commit/92eb3dfc9))

Exec approval matching is extracted into a dedicated `approvalMatchesSystemRunRequest` function with the `SystemRunApprovalBinding` type in a new `src/gateway/node-invoke-system-run-approval-match.ts` module. The function supports both argv array matching (via `commandArgv`) and legacy command-text matching for backward compatibility with existing approval records. Extracting the matching logic into a standalone module enables independent testing and provides a clear extension point for future approval precision improvements.

#### Unify IPv6 special-use blocking ([`61b3246a7`](https://github.com/openclaw/openclaw/commit/61b3246a7))

IPv6 special-use range blocking is consolidated into a single `isBlockedSpecialUseIpv6Address` function in `src/shared/net/ip.ts`. The previous `PRIVATE_OR_LOOPBACK_IPV6_RANGES` constant is renamed to `BLOCKED_IPV6_SPECIAL_USE_RANGES` to better reflect its broader scope beyond just private and loopback addresses. Test fixtures for blocked IPv6 literals are centralized in the new `src/shared/net/ip-test-fixtures.ts` file, replacing the previously fragmented `isPrivateOrLoopbackIpAddress` logic.

#### Block workspace hardlink alias escapes ([`04d91d031`](https://github.com/openclaw/openclaw/commit/04d91d031))

A hardlink bypass attack creates multiple directory entries pointing to the same inode, allowing a path inside the workspace to alias a file outside it. This commit introduces `assertNoHardlinkedFinalPath` in `src/infra/hardlink-guards.ts` that detects hardlinks by checking `stat.nlink > 1`. The guard is integrated into sandbox path resolution, workspace tools, and fs-bridge operations. Hardlink-target aliases are blocked by default with tests covering hardlink escapes through read, write, and patch operations.

#### Bind system.run approvals to argv identity ([`03e689fc8`](https://github.com/openclaw/openclaw/commit/03e689fc8))

Exec approval request payloads gain a `commandArgv` field in `src/infra/exec-approvals.ts` — an optional string array for precise argv-level matching. When this field is present in a stored approval, the incoming request's argv must match exactly — preventing shell injection via trailing-space or whitespace mutations in command tokens. `formatExecCommand` in `src/infra/system-run-command.ts` is updated to preserve whitespace in argv tokens (no trim), so whitespace-altered tokens produce a detectable mismatch. Approvals without a stored argv fall back to legacy command-text matching for backward compatibility with existing approval records.

#### Block IPv6 multicast SSRF bypass ([`baf656bc6`](https://github.com/openclaw/openclaw/commit/baf656bc6))

IPv6 multicast addresses (`ff00::/8`) are added to the SSRF blocklist in `src/shared/net/ip.ts`. Previously, multicast addresses such as `ff02::1` (all nodes) and bracketed URL variants could bypass the existing SSRF guards since the blocklist did not include the multicast range. The fix adds `ff00::/8` to `PRIVATE_OR_LOOPBACK_IPV6_RANGES` (before the range set was renamed by `61b3246a7`), blocking link-local multicast exfiltration via web-fetch.

#### Isolate account-scoped allowlist and pending requests ([`cf8d01bc5`](https://github.com/openclaw/openclaw/commit/cf8d01bc5))

The new `shouldIncludeLegacyAllowFromEntries` function in `src/pairing/pairing-store.ts` restricts legacy channel-level `AllowFrom` entries to the default account only. For non-default accounts, `readChannelAllowFromStore` and `readChannelAllowFromStoreSync` return only scoped entries, preventing cross-account implicit approvals via the backward-compatibility fallback. This ensures that multi-account deployments cannot inherit allowlist entries from another account's pairing history.

#### Fail closed group allowlist against DM pairing store ([`c7352f6b3`](https://github.com/openclaw/openclaw/commit/c7352f6b3))

Telegram group access decisions now use `normalizeAllowFrom` instead of `normalizeAllowFromWithStore` for `effectiveGroupAllow` computation in `buildTelegramMessageContext`. This severs the connection between DM pairing-store entries and group message authorization: previously, a paired DM partner could inadvertently satisfy a group allowlist check because the pairing store was consulted in the group allow-from resolution. Groups now fail closed — only explicit group `allowFrom` configuration grants access, and DM pairing entries cannot implicitly authorize group messages.

#### Harden approval-bound node exec cwd handling ([`f789f880c`](https://github.com/openclaw/openclaw/commit/f789f880c))

`hardenApprovedExecutionPaths` in `src/node-host/invoke-system-run.ts` validates the working directory for approval-bound node exec by comparing `lstat`, `stat`, and `realpath` results via `sameFileIdentity`. Symlink `cwd` values are blocked with `SYSTEM_RUN_DENIED`, and paths that fail identity comparison are rejected. The helper `isPathLikeExecutableToken` detects path-like argv tokens to guard against cwd TOCTOU symlink substitution attacks. This extends path escape prevention to approved exec invocations, which previously only had sandbox-level path guards.

#### Gate pin/reaction system events by sender auth ([`75dfb71e4`](https://github.com/openclaw/openclaw/commit/75dfb71e4))

Slack pin and reaction system events now call `authorizeSlackSystemEventSender` in `src/slack/monitor/auth.ts` before enqueuing for processing. Previously, Slack pins and reactions only verified the channel allowlist. The added sender check ensures both sender authorization and channel authorization are verified, preventing unauthorized users from triggering AI assistant responses via Slack pin/unpin or reaction events.

#### Remove dmPolicy from group allow context helper ([`470c606da`](https://github.com/openclaw/openclaw/commit/470c606da))

`resolveTelegramGroupAllowFromContext` in `src/telegram/bot/helpers.ts` no longer accepts a dmPolicy parameter. Groups no longer inherit the DM delivery policy in their allowlist context resolution, completing the separation of group and DM authorization paths initiated by `c7352f6b3`. A code comment at the function's `effectiveGroupAllow` assignment explicitly documents that DM pairing store entries are not a group authorization source.
