#### Allow RFC2544 benchmark range (198.18.0.0/15) through SSRF filter ([`9df80b73e`](https://github.com/openclaw/openclaw/commit/9df80b73e))

Telegram's API and file servers resolve to IPs in the 198.18.0.0/15 range (RFC 2544 benchmarking range). The SSRF guard was blocking these addresses because `ipaddr.js` classifies them as `reserved`, and an explicit `RFC2544_BENCHMARK_PREFIX` check blocked them unconditionally. This commit exempted 198.18.0.0/15 from the `isBlockedSpecialUseIpv4Address()` reserved-range check. Other reserved ranges (TEST-NET-2, TEST-NET-3, documentation prefixes) remain blocked. A follow-up commit (`3af9d1f8e`) immediately tightened this by making the exemption policy-opt-in only, restoring the default-block posture.

#### Scope Telegram RFC2544 SSRF exception to policy opt-in (#24982) (thanks @stakeswky) ([`3af9d1f8e`](https://github.com/openclaw/openclaw/commit/3af9d1f8e))

The unconditional 198.18.0.0/15 exemption from `9df80b73e` was correctly identified as too broad — it would allow any SSRF policy to reach RFC2544 benchmark IPs. This commit reversed the unconditional exemption: 198.18.0.0/15 is blocked by default again. Access is only permitted when a caller explicitly sets `allowRfc2544BenchmarkRange: true` in the `SsrfPolicy` object. The Telegram media download path already sets this flag, so Telegram functionality is unaffected. All other URL fetch paths remain blocked for RFC2544 addresses.

#### Allowlist api.telegram.org in media SSRF policy ([`dd14daab1`](https://github.com/openclaw/openclaw/commit/dd14daab1))

Added `allowedHostnames: ["api.telegram.org"]` to the `TELEGRAM_MEDIA_SSRF_POLICY` constant used for Telegram file downloads. This allows the SSRF guard to trust `api.telegram.org` by hostname even when DNS/proxy resolution maps it to private or reserved IP ranges (such as the 198.18.0.0/15 range in restricted networks). The policy constant already carried `allowRfc2544BenchmarkRange: true`; this commit adds hostname-level trust as a second layer, reducing reliance on IP-range exemptions.

#### GroupAllowFrom sender filter bypassed when groupPolicy is allowlist ([`c6bb7b0c0`](https://github.com/openclaw/openclaw/commit/c6bb7b0c0))

A logic error in `resolveChannelGroupPolicy()` allowed a `senderFilterBypass` path when `groupPolicy` was `"allowlist"`, no groups were configured (`!hasGroups`), and the request carried a `hasGroupAllowFrom` flag. This bypass could grant access to senders who should have been rejected by the allowlist. The fix corrects the `senderFilterBypass` boolean so the bypass is no longer reachable under allowlist mode when no groups are configured. Related to GHSA-f6h3-846h-2r8w.

#### Redact phone numbers and message content from WhatsApp logs ([`aef45b2ab`](https://github.com/openclaw/openclaw/commit/aef45b2ab))

WhatsApp outbound and heartbeat-runner paths were logging JIDs and phone numbers in plaintext. This commit applies `redactIdentifier()` (SHA-256 one-way hash) to JIDs and phone numbers before they reach log output, replacing them with `[redacted:<hash_prefix>]` tokens. Developers can still correlate log entries for the same identifier without recovering the original value. The `redactIdentifier` helper is imported from `src/web/logging/redact-identifier.js`.

#### Sanitize pairing recovery requestId hints (#24771) (thanks @markmusson) ([`69a541c3f`](https://github.com/openclaw/openclaw/commit/69a541c3f))

The pairing recovery flow in `src/commands/status.command.ts` extracted a `requestId` hint from URL query parameters and displayed it in CLI output without validation. A malicious pairing URL could inject arbitrary characters. This commit adds `sanitizeRequestId()`, a strict allowlist validator that accepts only strings matching `/^[A-Za-z0-9][A-Za-z0-9._:-]{0,127}$/` and returns `null` for any input that doesn't conform. The CLI display path uses the sanitized value, eliminating the injection vector.

#### Consume allow-once approvals to prevent replay ([`3f5e7f815`](https://github.com/openclaw/openclaw/commit/3f5e7f815))

Gateway exec approvals with `decision: "allow-once"` were not atomically consumed on use. An attacker with access to the same approval record ID could replay the approval to authorize additional `system.run` executions. This commit adds `consumeAllowOnce()` to `src/gateway/exec-approval-manager.ts`, which atomically clears the `decision` field to `undefined` on first use and returns `false` on subsequent calls. The `system.run` approval check at `src/gateway/node-invoke-system-run-approval.ts` now calls `consumeAllowOnce()` and rejects the request if it returns `false`.

#### Pass SSRF policy for localhost attachment downloads ([`dd41a7845`](https://github.com/openclaw/openclaw/commit/dd41a7845))

BlueBubbles is an optional plugin that downloads message attachments from a locally-running BlueBubbles server (typically `localhost` or a private LAN address). The attachment download path was not passing any SSRF policy, which meant the SSRF guard blocked all private-network requests even when the user explicitly configured `allowPrivateNetwork: true` in the plugin config. This commit threads the `allowPrivateNetwork` flag from the resolved account through `resolveBlueBubblesServerAccount()` and passes it as `ssrfPolicy: { allowPrivateNetwork: true }` to attachment downloads. Without the explicit policy, attachment fetches would have silently failed for any gateway with strict SSRF enforcement.

#### Respect selfChatMode config in access-control ([`1e23d2ece`](https://github.com/openclaw/openclaw/commit/1e23d2ece))

The `selfChatMode` config field in WhatsApp account settings controls whether the bot accepts messages sent to itself (self-chat). Although `accounts.ts` resolved this field, it was never consumed in `src/web/inbound/access-control.ts` — the access-control logic only used the `isSelfChatMode()` heuristic (which infers self-chat from `allowFrom` config). An explicit `selfChatMode: false` in config had no effect, leaving a misconfiguration window where self-chat could not be disabled. The fix applies nullish coalescing: `account.selfChatMode ?? isSelfChatMode(params.selfE164, configuredAllowFrom)`, so an explicit config value takes precedence over the heuristic.

- **Privacy hardening (new control):** `aef45b2ab` applies `redactIdentifier()` SHA-256 hashing to JIDs and phone numbers in WhatsApp log output, reducing PII exposure in logs.
