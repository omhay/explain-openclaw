### Telegram Group Callback Auth Delegation ([`58fef1d70`](https://github.com/openclaw/openclaw/commit/58fef1d70))

Changes `enforceGroupAllowlistAuthorization` from `true` to `false` for `"callback-allowlist"` callback actions in group chats. Previously, both the upstream `shouldSkipGroupMessage()` check (which enforces group policy + allowlist) and a second allowlist gate at the callback stage had to pass. The second gate was blocking users whose original command had already been authorized. Now group auth is delegated entirely to `shouldSkipGroupMessage()`.

**Audit 2 Claim 5 relation:** This is a behavioral change to group callback authorization. The upstream `shouldSkipGroupMessage()` check remains enforced; removing the redundant second gate does not weaken the overall auth model.

### Config Credential Preservation During Merge ([`c289b5ff9`](https://github.com/openclaw/openclaw/commit/c289b5ff9))

Updates `ensureOpenClawModelsJson()` to preserve existing `apiKey` and `baseUrl` fields when merging new provider configs in `merge` mode. Previously, a simple object spread (`{ ...existingProviders, ...providers }`) could overwrite agent-level credentials with incoming values that lack those fields. The fix explicitly loops through provider entries and re-applies `apiKey`/`baseUrl` from the existing config if they are present before writing the merged `models.json`.

**Audit 1 Claim 1 relation:** Improves credential integrity by preventing accidental overwrite of agent-scoped credentials during config merges. Tangential to the plaintext-storage issue: credentials remain in `models.json` with `0o600` permissions; this commit ensures they are not silently lost during provider syncs.

### Single-Account Channel Config Migration ([`dfa0b5b4f`](https://github.com/openclaw/openclaw/commit/dfa0b5b4f))

Adds `shouldMoveSingleAccountChannelKey()` and `moveSingleAccountChannelSectionToDefaultAccount()` to `src/channels/plugins/setup-helpers.ts`. When a channel has no existing multi-account structure, credentials and config fields (`token`, `botToken`, `webhookSecret`, `accessToken`, `password`, `signalNumber`, etc.) from a defined allowlist of 23 keys are cloned via `structuredClone()` into an `accounts.default` sub-object. `doctor-config-flow.ts` and onboarding helpers invoke this migration when promoting single-account channels to the multi-account structure.

**Audit 2 Claim 6 relation:** Structural migration of credential fields into scoped sub-objects. Does not address the pipe-delimited token format itself but reduces the surface where top-level credential keys are exposed directly in the channel config root.
