#### Fail closed on empty group allowFrom override ([`6bc7544a6`](https://github.com/openclaw/openclaw/commit/6bc7544a6))

`evaluateTelegramGroupBaseAccess()` now returns `{ allowed: false, reason: "group-override-unauthorized" }` when the per-group/topic `allowFrom` override list is present but has no entries (`!params.effectiveGroupAllow.hasEntries`). Previously, an empty override list would fall through to the `requireSenderForAllowOverride` check, which could allow messages if the sender was already known. The new guard is inserted before that check, ensuring an explicitly empty group allowlist fails closed rather than open.

#### Harden Slack file-only fallback placeholder (#25181) (thanks @justinhuangcode) ([`b247cd6d6`](https://github.com/openclaw/openclaw/commit/b247cd6d6))

The filename mapping in `prepareSlackMessage()` changed from `.map((f) => f.name ?? "file")` to `.map((f) => f.name?.trim() || "file")`. The old form passed through whitespace-only filenames unchanged (since a string of spaces is truthy and `??` only catches `null`/`undefined`). The new form trims whitespace before testing truthiness, so blank or whitespace-only filenames are replaced with the safe `"file"` sentinel. This prevents a whitespace-only filename from being forwarded to the agent inside a `[Slack file: ]` placeholder.

#### Deliver file-only messages when all media downloads fail ([`def28a87b`](https://github.com/openclaw/openclaw/commit/def28a87b))

When a Slack message contains only file attachments and every download fails, `resolveSlackMedia` returns null, leaving `mediaPlaceholder` undefined. Previously `rawBody` would be empty and `prepareSlackMessage()` would silently drop the message. This commit adds a `fileOnlyFallback` path: if `mediaPlaceholder` is absent but `message.files` is non-empty, the first up-to-five filenames are joined as `[Slack file: name1, name2]` and appended to `rawBody`. The fallback uses `f.name ?? "file"` for each entry (hardened to `.trim() || "file"` in the follow-up `b247cd6d6`), ensuring the agent always receives a delivery signal even when file downloads fail completely.

#### Note narrow filesystem roots for tool access ([`8f5f599a3`](https://github.com/openclaw/openclaw/commit/8f5f599a3))

A one-line hardening note was added to the official security documentation (`docs/gateway/security/index.md`) advising operators to keep agent workspace and sandbox roots narrow. Broad roots (such as `~/`) can expose sensitive local state — for example files under `~/.openclaw` — to agent filesystem tools even when no path traversal exploit is present. This directly addresses Audit 1 Claim 6, which flagged the risk of broad path roots in agent-accessible directories. The change is documentation-only and does not modify any runtime control.

#### Resolve wrong provider prefix when session has model without modelProvider ([`177386ed7`](https://github.com/openclaw/openclaw/commit/177386ed7))

A new `normalizeSessionRuntimeModelFields()` function in `src/config/sessions/types.ts` canonicalizes the `model` and `modelProvider` fields in session entries. When `model` is empty or whitespace-only, both `model` and `modelProvider` are removed entirely. When `model` is valid but `modelProvider` is empty or absent, `modelProvider` is removed (preventing the TUI from constructing a malformed `<provider>/<model>` prefix from a blank provider string). The function is called from `normalizeSessionStore()` in `src/config/sessions/store.ts`, so it runs on every session load, ensuring legacy entries are cleaned on first access. This prevents the TUI from displaying an incorrect provider prefix, which also affects the cron isolated-agent runner that reads session model fields.
