### DM Allowlist Isolation ([`8bdda7a65`](https://github.com/openclaw/openclaw/commit/8bdda7a65))

Hardens the separation between DM pairing allowlists and group channel authorization. Previously, the `allowFrom` lists associated with DM pairing could inadvertently influence group channel auth decisions — allowing a DM-paired user to gain access to group channels without an explicit group allowlist entry.

The fix refactors `resolveEffectiveAllowFromLists()` in `dm-policy-shared.ts` to strictly separate DM and group sources. New helpers `mergeDmAllowFromSources()` and `resolveGroupAllowFromSources()` in `allow-from.ts` ensure each auth context reads only the lists appropriate to its channel type. `resolveDmGroupAccessDecision()` and `resolveDmGroupAccessWithLists()` are updated to enforce this split end-to-end, including in the Mattermost extension and the Line and Telegram bots.

**Audit 1 Claim 5 / Audit 2 Claim 5 relation:** Prevents DM pairing allowlists from leaking into group channel authorization, a concrete RBAC boundary enforcement fix.

### Plugin Channel Canonicalization ([`0ed675b1d`](https://github.com/openclaw/openclaw/commit/0ed675b1d))

Introduces canonical path normalization before plugin channel auth matching in `server-http.ts`. Four new functions gate the protected-prefix check: `normalizeSecurityPath()` percent-decodes and normalizes slashes, `canonicalizePathForSecurity()` handles case normalization on the decoded path, `hasProtectedPluginChannelPrefix()` checks the canonical path against the protected prefix list, and `isProtectedPluginChannelPath()` combines both steps for the authorizer.

Previously, the auth check operated on the raw request pathname. A URL-encoded prefix (e.g., `/%2Fplugin/...`) or mixed-case variation could bypass the prefix guard and reach protected plugin routes without proper authorization.

**Audit 2 Claim 3 relation:** Closes a potential path canonicalization bypass for plugin channel routes, strengthening the gateway auth layer.

### Slack System Event Auth Gates ([`3d30ba18a`](https://github.com/openclaw/openclaw/commit/3d30ba18a))

All Slack system events — `member_joined_channel`, `member_left_channel`, `message_changed`, `message_deleted`, and `thread_broadcast` — now pass through `authorizeAndResolveSlackSystemEventContext()` before any processing. This function verifies sender allowlists, resolves the correct session key, and returns `null` (causing an early return) if the event does not pass the auth check.

Previously, these event handlers in `registerSlackMemberEvents()` and `registerSlackMessageEvents()` could process events from unauthorized senders. The commit also adds `subtype` filtering for `message_changed` and `message_deleted` events to reject bot-generated edits that do not originate from an authorized user.

**Audit 1 Claim 7 relation:** Extends webhook/event authorization to previously ungated Slack system event types.

### Exec Approval Channel-Source Tracking ([`da0ba1b73`](https://github.com/openclaw/openclaw/commit/da0ba1b73))

Adds channel-source tracking to exec approval requests. `ExecApprovalRequestPayload` gains four new optional fields: `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId`. These fields record which messaging channel and recipient triggered the command that requires approval.

The exec approval forwarder in `exec-approval-forwarder.ts` uses these fields to route approval notifications back to the exact channel and thread that originated the request, rather than falling back to the session's last-known delivery target. Server-side HTTP path hardening in `server-http.ts` also ships in this commit, adding `isProtectedPluginChannelPath()` to the gateway authorizer for completeness.

**Audit 2 Claim 5 relation:** Improves exec approval attribution and routing accuracy, reducing the risk of approval notifications being delivered to the wrong channel in multi-account setups.

### Web SSRF Guard Unification ([`b74be2577`](https://github.com/openclaw/openclaw/commit/b74be2577)) — ESCALATED

Creates a new module `web-guarded-fetch.ts` that centralizes the SSRF guard for all web tool HTTP requests. The module exports three items: `WEB_TOOLS_TRUSTED_NETWORK_SSRF_POLICY` (the shared SSRF policy constant), `fetchWithWebToolsNetworkGuard()` (guarded fetch with hardcoded `proxy: "env"` enforcement), and `withWebToolsNetworkGuard()` (a wrapper for fetch sequences that need the full connection lifecycle).

Previously, web-fetch and web-search had separate (and potentially diverging) SSRF enforcement paths. This escalation is flagged because consolidating into a single guarded path eliminates a class of guard-bypass risk that could arise if either tool's path evolved independently. The hardcoded `proxy: "env"` ensures the environment-configured HTTP proxy is always honored for web tool requests.

**Audit 2 Claim 4 relation:** Closes a structural gap where separate fetch paths could diverge in SSRF policy enforcement, a defense-in-depth improvement over per-tool guard application.

### Boundary-Safe File Reads ([`242188b7b`](https://github.com/openclaw/openclaw/commit/242188b7b))

Introduces `boundary-file-read.ts`, a new module providing boundary-enforced file reading for workspace bootstrap and config includes. Three exports: `canUseBoundaryFileOpen()` (checks that the `ioFs` object supports all required syscalls), `openBoundaryFileSync()` (synchronous boundary-checked open with O_NOFOLLOW semantics, symlink and hardlink rejection, and size limits), and `openBoundaryFile()` (async variant).

Workspace bootstrap file loading in `workspace.ts` (`loadWorkspaceBootstrapFiles()`) now uses `openBoundaryFile()` instead of a plain `fs.readFile()`. The sandbox fs-bridge also adopts `openBoundaryFile()`. This means any attempt to plant a symlink pointing outside the workspace directory into a bootstrap `.md` file will be rejected at read time.

**Audit 1 Claims 5+6 relation:** Prevents symlink-based escape from workspace file reads, addressing a structural gap in the bootstrap pipeline's path validation.

### Web Proxy Path Fix ([`46003e85b`](https://github.com/openclaw/openclaw/commit/46003e85b))

Completes the proxy-guarded path unification by migrating `web-search.ts` and `web-fetch.ts` to call `fetchWithWebToolsNetworkGuard()` and `withWebToolsNetworkGuard()` from the `web-guarded-fetch.ts` module (introduced by `b74be2577`). Also fixes the Brave Search API not honoring the `HTTP_PROXY` environment variable, which was a bug introduced when the search tool's fetch path diverged from the main web-fetch implementation.

The internal guard in `fetch-guard.ts` is updated to support the unified policy model.

**Audit 2 Claim 4 relation:** Ensures all web tool outbound fetches use the shared SSRF guard, eliminating the divergent-path risk identified in the escalation of `b74be2577`.

### Config Include Path Hardening ([`9925ac6a2`](https://github.com/openclaw/openclaw/commit/9925ac6a2)) — GHSA-56pc-6hvp-4gv4

Hardens the `$include` directive in config loading to use `openBoundaryFileSync()` from `boundary-file-read.ts`. `readConfigIncludeFileWithGuards()` in `includes.ts` checks `canUseBoundaryFileOpen()` and, when supported, routes all include-file reads through the boundary-checked open. On platforms where the boundary open is not supported (checked via `canUseBoundaryFileOpen()`), a `safeRealpath()` fallback is used.

This patches GHSA-56pc-6hvp-4gv4 (arbitrary file read via `$include`). An attacker with write access to the config directory could previously craft a `$include` pointing to an arbitrary path (e.g., `/etc/passwd` or SSH keys). The boundary check now rejects paths outside the config root, symlinks, and hardlinks before `readFileSync()` is called.

**Audit 1 Claim 6 relation:** Directly closes the path traversal vector for config includes. GHSA-56pc-6hvp-4gv4 status: **patched** by this commit and `242188b7b`.
