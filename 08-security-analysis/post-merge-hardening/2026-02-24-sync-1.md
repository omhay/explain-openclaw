#### Gateway: harden cron.runs jobId path handling (openclaw#24038) thanks @Takhoffman ([`259d86335`](https://github.com/openclaw/openclaw/commit/259d86335))

Adds `assertSafeCronRunLogJobId()` in `src/cron/run-log.ts` that validates jobId values before they are used in log path construction. The function rejects characters that would enable directory traversal (forward slashes, backslashes, null bytes) and enforces a minimum length. `resolveCronRunLogPath()` now passes jobId through this validator and performs a post-resolution boundary check (`resolvedPath.startsWith(storePath)`) to prevent traversal even after path normalization.

At the gateway protocol layer, `CronRunLogJobIdSchema` (in `src/gateway/protocol/schema/cron.ts`) enforces the same restriction with a `^[^/\\]+$` regex pattern, rejecting malformed jobId inputs at the API boundary before they reach the file system. This two-layer defense closes the path traversal vector against cron run log storage (Audit 1 Claim 6).

#### Gateway/Chat UI: sanitize untrusted wrapper markup in final payloads ([`a10ec2607`](https://github.com/openclaw/openclaw/commit/a10ec2607))

Exports a new `stripEnvelopeFromMessage()` function (single-message variant) in `src/gateway/chat-sanitize.ts` alongside the existing `stripEnvelopeFromMessages()`. The `broadcastChatFinal()` function in `src/gateway/server-methods/chat.ts` now calls `stripEnvelopeFromMessage()` before passing to `stripInlineDirectiveTagsFromMessageForDisplay()`, ensuring that UNTRUSTED_CONTEXT / EXTERNAL_UNTRUSTED_CONTENT envelope markup is removed from chat payloads before delivery to clients. Prevents attacker-supplied metadata embedded in wrapper content from reaching the chat UI.

#### Fallback docker user to workspace owner uid/gid ([`a30f9c867`](https://github.com/openclaw/openclaw/commit/a30f9c867))

The function validates that the returned uid/gid are non-negative integers before applying them.

#### Harden safe-bin trust and add explicit trusted dirs ([`64b273a71`](https://github.com/openclaw/openclaw/commit/64b273a71))

Removes implicit PATH-based trust from safe-bin execution. The refactored `getTrustedSafeBinDirs()` no longer accepts a `pathEnv` parameter and no longer auto-trusts directories from the runtime PATH environment — only explicit `baseDirs` and `extraDirs` parameters are trusted. Adds a new `safeBinTrustedDirs` configuration field for user-defined trusted directory allowlists.

A new `renderSafeBinSegmentArgv()` function in `src/infra/exec-approvals-analysis.ts` pins resolved absolute executable paths in shell command reconstruction, preventing PATH shadowing attacks where a directory appearing earlier in PATH could override a trusted executable. `normalizeTrustedSafeBinDirs()` in `src/config/normalize-exec-safe-bin.ts` normalizes and deduplicates trusted directory lists in both config I/O and runtime policy. Together these changes close the implicit PATH trust vector that enabled PATH shadowing attacks against safe-bin execution.

#### Strip hidden content to prevent indirect prompt injection ([`44727dc3a`](https://github.com/openclaw/openclaw/commit/44727dc3a))

Adds a new `src/agents/tools/web-fetch-visibility.ts` module with two exported functions. `sanitizeHtml()` parses fetched HTML and removes elements hidden via CSS (`display: none`, `visibility: hidden`, `opacity: 0`, `font-size: 0`, color-transparent, and negative text-indent), class names (`sr-only`, `d-none`, `hidden`, `invisible`), or `aria-hidden` attributes. `stripInvisibleUnicode()` strips zero-width characters and other invisible Unicode codepoints.

Both functions are called in `src/agents/tools/web-fetch-utils.ts` during the markdown conversion pipeline. This prevents indirect prompt injection attacks where malicious web pages embed hidden instructions (invisible to human readers but visible to the LLM) in fetched content.

#### Add SSRF protection to attachment downloads via redirect and DNS validation ([`26644c4b8`](https://github.com/openclaw/openclaw/commit/26644c4b8))

Adds three-layer SSRF protection to the MS Teams attachment download flow in `fetchWithAuthFallback()` (in `extensions/msteams/src/attachments/download.ts`):

1. `safeFetch()` (in `extensions/msteams/src/attachments/shared.ts`) uses `redirect: "manual"` to intercept redirects and validates each redirect destination against the hostname allowlist and DNS-resolved IP before following it.
2. `resolveAndValidateIP()` performs DNS resolution and checks the result against `isPrivateOrReservedIP()`, blocking redirects that resolve to private, loopback, or reserved IP ranges.
3. All redirect hops are validated — not just the initial URL — preventing multi-hop DNS rebinding where an allowlisted hostname redirects through an intermediate host to a private IP.

Addresses Audit 2 Claim 4 (DNS rebinding SSRF) for the MS Teams channel attachment download path.

#### Block shell-wrapper line-continuation allowlist bypass ([`3f0b9dbb3`](https://github.com/openclaw/openclaw/commit/3f0b9dbb3))

Adds `hasShellLineContinuation()` guard in `src/infra/exec-approvals-allowlist.ts` that calls `analysisFailure()` (returning `{analysisOk: false, allowlistSatisfied: false}`) when a command contains backslash-newline sequences. Shell line-continuation semantics are shell-dependent and can rewrite token boundaries at runtime, enabling bypass of pattern-based allowlist checks via embedded newlines that the allowlist parser does not see.

`isEscapedLineContinuation()` in `src/infra/exec-approvals-analysis.ts` adds the same guard at the token-parsing layer, returning `{ok: false, reason: "unsupported shell token: newline"}` for backslash-escaped newlines. This closes the line-continuation bypass vector in both the allowlist analysis and command token parser (Audit 2 Claim 7 — shell injection via incomplete regex).
