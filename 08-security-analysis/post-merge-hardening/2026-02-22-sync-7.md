### Key Security Changes

1. **`4550a5200`** — **TUI Model Picker: Enforce Config-Driven Allowlist:**

   The `models.list` gateway handler (`src/gateway/server-methods/models.ts`) previously returned the raw model catalog unconditionally to all callers — including the TUI model picker. This allowed users to select models that an operator had intended to restrict via the allowlist config.

   The fix imports `buildAllowedModelSet()` (`src/gateway/server-methods/models.ts:2`) and calls it before returning the catalog:

   - `src/gateway/server-methods/models.ts:2` — `buildAllowedModelSet` imported from `../../agents/model-selection.js`
   - `src/gateway/server-methods/models.ts:26` — `loadGatewayModelCatalog()` still fetches the full raw catalog
   - `src/gateway/server-methods/models.ts:28` — `allowedCatalog` = filtered result via `buildAllowedModelSet()`
   - `src/gateway/server-methods/models.ts:33` — fallback: if allowlist is empty, returns full catalog (behavior unchanged when no allowlist configured)

   This aligns the `models.list` path with the same filtering already applied in `agent.ts` and `model-picker.ts`.

   **Hardening note:** The fallback (`allowedCatalog.length > 0 ? allowedCatalog : catalog`) silently falls back to exposing all models when the allowlist config is empty. Operators intending model restriction must configure at least one allowed model.

   **Relevance to audit claims:** Addresses **Audit 2 Claim 5** (self-approving agent / no RBAC — restricts which models are visible and selectable in TUI, consistent with operator policy).

2. **`3317b49d3`** — **QMD Memory Search: mcporter Keep-Alive Backend (Defense-in-Depth Concern):**

   Introduces optional routing of QMD memory searches through the `mcporter` MCP runtime (external process) rather than spawning a `qmd` process per query. Enabled via a new `memory.qmd.mcporter` config block.

   New types and validation:
   - `src/config/zod-schema.ts:75` — `MemoryQmdMcporterSchema` validates the new config block (verified)
   - `src/memory/backend-config.ts:54` — `ResolvedQmdMcporterConfig` type (verified)
   - `src/memory/backend-config.ts:254` — `resolveMcporterConfig()` resolves the config; applies only `.trim()` to `serverName` (verified)

   New `qmd-manager.ts` functions:
   - `src/memory/qmd-manager.ts:899` — `ensureMcporterDaemonStarted()` auto-spawns `mcporter daemon` on first search when `startDaemon: true`
   - `src/memory/qmd-manager.ts:993` — `runQmdSearchViaMcporter()` performs the search via MCP tool call
   - `src/memory/qmd-manager.ts:1004` — MCP tool-call selector: `` `${serverName}.${tool}` `` — `serverName` comes directly from config

   **Security concern:** The user-supplied `serverName` config value receives only `.trim()` normalization (`backend-config.ts:262`) before embedding in the MCP tool-call selector. No validation against shell metacharacters, path-separators, or dot injection is performed. If an attacker can write the config, a crafted `serverName` could redirect MCP tool calls to an unintended server. The `startDaemon: true` default also auto-spawns `mcporter daemon` on first search, expanding the external process attack surface.

   **Defense-in-depth gap:** `serverName` should be validated against a strict allowlist pattern (alphanumeric + hyphens, no dots or shell metacharacters) before use in MCP selectors.

   **Relevance to audit claims:** Relates to **Audit 2 Claim 1** (config injection — indirect vector via `serverName` in MCP selector rather than `setupCommand` shell execution, but same trust model).

3. **`d3991d6aa`** + **`0bb81f729`** — **Sandbox Media Path Validation Hardening (Escalated from Tier B):**

   Two commits in the sandbox media validation chain, both escalated to security due to production file changes in `src/agents/sandbox-paths.ts`.

   **`0bb81f729`** fixes `resolveSandboxedMediaSource()` to explicitly allow `os.tmpdir()` paths that had been incorrectly rejected alongside sandboxed paths. Adds:
   - `src/agents/sandbox-paths.ts:94` — `tmpDir = path.resolve(os.tmpdir())`
   - `src/agents/sandbox-paths.ts:96` — `isPathInside(tmpDir, resolved)` check allows tmpdir paths through

   **`d3991d6aa`** hardens the same function by adding symlink-escape validation for temp dir paths:
   - `src/agents/sandbox-paths.ts:97` — `assertNoSymlinkEscape(resolved)` call added for tmpdir paths
   - `src/agents/sandbox-paths.ts:108` — `assertNoSymlinkEscape()` function definition (resolves each path component, rejects symlinks pointing outside the validated root)

   Together these commits close a path where `resolveSandboxedMediaSource()` would either incorrectly reject valid tmpdir media (before `0bb81f729`) or accept symlink-traversed tmpdir paths (before `d3991d6aa`).

   **Relevance to audit claims:** Addresses **Audit 2 Claim 3** (sandbox path traversal / symlink escape in media handling). Strengthens the tmpdir media validation chain.
