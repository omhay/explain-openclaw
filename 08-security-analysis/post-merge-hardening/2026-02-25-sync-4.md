#### Detect Telnyx webhook replay ([`a3c4f56b0`](https://github.com/openclaw/openclaw/commit/a3c4f56b0))

Adds replay detection to Telnyx webhook verification. The `verifyTelnyxWebhook()` function in `extensions/voice-call/src/webhook-security.ts` now checks incoming webhook signatures against a cache of recently seen signature keys, setting `isReplay: true` in the `TelnyxVerificationResult` when a replayed request is detected. The webhook handler rejects replayed requests before processing. This closes a vector where a valid but previously seen webhook could be delivered twice — relevant to Audit 1 Claim 7 (webhook signature bypass) by ensuring freshness alongside signature validity.

#### Simplify DM media auth precheck flow ([`36d1e1dcf`](https://github.com/openclaw/openclaw/commit/36d1e1dcf))

Refactors Telegram sender identity resolution to reduce access control logic duplication. The new helper `resolveTelegramSenderIdentity()` in `src/telegram/dm-access.ts` normalizes sender attributes (username, userId, firstName, lastName, candidateId) into a consistent type. The `enforceTelegramDmAccess()` function is updated to use this helper, centralizing the identity data used in allowlist matching. This reduces the risk of inconsistent identity handling across DM access checks, hardening the authorization surface against identity-resolution bypass.

#### Classify hook sessions case-insensitively ([`9924f7c84`](https://github.com/openclaw/openclaw/commit/9924f7c84))

Hardens `isExternalHookSession()` and `getHookType()` in `src/security/external-content.ts` to normalize input via `trim().toLowerCase()` before prefix matching. Previously, an attacker who controlled the session identifier format could potentially use uppercase or whitespace-padded values to bypass external content security classification (which gates prompt injection defenses and trust-level checks). New tests verify that `HOOK:gmail:msg-123` and whitespace-padded variants like `"  HOOK:webhook:123  "` are correctly classified as external sessions.

#### Harden safe-bin trusted dir diagnostics ([`4355e0826`](https://github.com/openclaw/openclaw/commit/4355e0826))

Introduces runtime diagnostics for safe-bin trusted directory security in `src/infra/exec-safe-bin-trust.ts` and `src/infra/exec-safe-bin-runtime-policy.ts`. Three new helpers provide defense-in-depth:

- `listWritableExplicitTrustedSafeBinDirs()`: detects explicitly configured trusted directories that are group- or world-writable, logging runtime warnings when found.
- `getTrustedSafeBinDirs()`: resolves the final set of trusted directories with caching and refresh support.
- `isTrustedSafeBinPath()`: checks whether a given binary path falls within a trusted directory, used during exec approval checks in `src/infra/exec-approvals-allowlist.ts`.

Doctor diagnostics also flag binaries that resolve outside trusted directories. This prevents privilege escalation where a trusted directory becomes writable by unprivileged processes.

#### Land hard breakage cluster from reviewed PR bases ([`e7a5f9f4d`](https://github.com/openclaw/openclaw/commit/e7a5f9f4d))

**Discord reasoning suppression (Audit 2 Claim 5 adjacent):** An `isReasoning` flag is applied to Claude's internal reasoning/thinking payloads to prevent them from being forwarded to external Discord delivery. Internal channels still receive full reasoning output; external surfaces see only the final response.

#### Block container namespace joins by default ([`14b6eea6e`](https://github.com/openclaw/openclaw/commit/14b6eea6e))

Implements default blocking of Docker `network: "container:<id>"` namespace joins across all sandbox types. Container namespace joins allow a container to share the network stack of another container — a significant isolation escape vector relevant to Audit 1 Claim 6 (path traversal in agent dirs via namespace access).

**Key changes:**
- Config schema adds `dangerouslyAllowContainerNamespaceJoin?: boolean` to the Docker sandbox config type. The option must be explicitly set to `true` to permit container namespace joins — a break-glass override that requires operator intent.
- Zod schema validation in `zod-schema.agent-runtime.ts` rejects both docker and browser sandbox network modes that request container namespace joins unless the override is set.
- `validateNetworkMode()` in `src/agents/sandbox/validate-sandbox-security.ts` enforces the policy at runtime during sandbox creation.
- Security audit in `src/security/audit-extra.sync.ts` flags both `"host"` and `"container:*"` modes as critical findings.
- Six documentation files updated to document the default-deny policy and the break-glass override.

Error messages specify: container namespace joins "bypass sandbox network isolation" and the override "should only be enabled when you fully trust this runtime."

#### Security: sanitize inherited host exec env ([`48b052322`](https://github.com/openclaw/openclaw/commit/48b052322))

Core implementation of host exec environment sanitization (Audit 2 Claim 8). The `sanitizeHostExecEnv()` function in `src/infra/host-env-security.ts` is now called from `src/node-host/invoke.ts` with `blockPathOverrides: true` when constructing the environment for bash tool execution. This strips dangerous inherited environment variables — including LD_PRELOAD, DYLD_INSERT_LIBRARIES, NODE_OPTIONS, and PATH overrides — from the process environment before passing it to exec. Previously, a malicious actor with control over the gateway's startup environment could inject these variables to hijack library loading or Node.js options on bash tool invocations. This commit is the core enforcement fix; `43a3ff3be` documents it in the changelog.

#### Restrict default safe-bin trusted dirs ([`b67e600bf`](https://github.com/openclaw/openclaw/commit/b67e600bf))

Narrows the default set of trusted directories used by the safe-bin exec allowlist in `src/infra/exec-safe-bin-runtime-policy.ts`. Previously, the default trusted directory list included paths that may not be exclusively controlled by the openclaw operator. By reducing the default trusted set, binaries in those directories are no longer auto-approved by the allowlist mechanism — they require explicit operator configuration. This reduces the blast radius of binary shadowing attacks (e.g., placing a malicious binary in a trusted directory). Documentation in `docs/tools/exec-approvals.md` and `docs/tools/exec.md` is updated to reflect the narrowed defaults.

#### Block svg data image opens and harden tests ([`e9750104b`](https://github.com/openclaw/openclaw/commit/e9750104b))

Blocks `data:image/svg+xml` URLs from being opened via the external URL handler in `ui/src/ui/open-external-url.ts`.
