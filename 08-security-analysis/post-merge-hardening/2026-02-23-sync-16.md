#### Fail Fast Exec Approvals When No Approvers Are Reachable ([`d24f5c1e3`](https://github.com/openclaw/openclaw/commit/d24f5c1e3))

`ExecApprovalManager` gains a fail-fast path: a new `hasExecApprovalClients()` probe is added to `GatewayRequestContext` (gated behind the operator-only scope check), and when the probe returns `false`, approval requests are immediately expired rather than lingering indefinitely. `ExecApprovalForwarder.handleRequested()` now returns `boolean` to signal forwarding success, enabling the server to detect and expire unforwardable requests at the point of delivery.

#### Restore Configurable Subagent Announce Timeout ([`320cf8eb3`](https://github.com/openclaw/openclaw/commit/320cf8eb3))

Restores user-facing subagent announce timeout configuration via the `agents.defaults.subagents.announceTimeoutMs` config field. A new `resolveSubagentAnnounceTimeoutMs()` helper validates and clamps the configured value between 1ms and `MAX_TIMER_SAFE_TIMEOUT_MS` to prevent timer overflow; `sendAnnounce()` and `sendSubagentAnnounceDirectly()` now use the resolved timeout instead of the previous hardcoded 15 seconds, preventing timeouts from misconfiguring the announce delivery path.

#### Harden Exec Approval Forwarding Target and Delivery ([`51b0772e1`](https://github.com/openclaw/openclaw/commit/51b0772e1))

Adds `resolveChannelAccountConfig()`, a generic case-insensitive account lookup helper used in exec approval forwarding, preventing typo-caused config mismatches from silently skipping delivery targets. `shouldSkipDiscordForwarding()` is refined to skip Discord text-fallback only when a Discord-specific handler is enabled for the same account (the previous logic was over-broad and could suppress valid delivery paths). Approval resolution now includes the full `request` object in `ExecApprovalResolved` for downstream delivery context.

#### Honor Explicit Bind Mounts Over Workspace Defaults ([`6f895eb83`](https://github.com/openclaw/openclaw/commit/6f895eb83))

`buildSandboxCreateArgs()` gains an `includeBinds` parameter (defaults to `true`, set to `false` for browser isolation) and a new `appendCustomBinds()` helper that applies operator-configured bind mounts after workspace defaults. A `mountSourcePriority()` tiebreaker in the mount comparators ensures custom binds sort after workspace defaults at the same container path, enabling read-only overlay mounts to shadow more permissive workspace bind mounts as a defense-in-depth measure.

#### Require Sender-Only Matching for Elevated allowFrom ([`02772b029`](https://github.com/openclaw/openclaw/commit/02772b029))

Removes `params.ctx.To` and `stripSenderPrefix(params.ctx.To)` from the token set inside `isApprovedElevatedSender()`. Previously the recipient address (`To` field) was incorrectly included as a matchable token, meaning an attacker who controlled the recipient address could potentially satisfy elevated `allowFrom` entries. This fix constrains matching to sender-only fields (`SenderE164`, `From`).

#### Harden toolsBySender Sender-Key Matching ([`5547a2275`](https://github.com/openclaw/openclaw/commit/5547a2275))

Introduces a typed sender-key schema for `toolsBySender` configuration. `TOOLS_BY_SENDER_KEY_TYPES` and `ToolsBySenderKeyType` (in `types.tools.ts`) define explicit key prefixes (`id:`, `e164:`, `username:`, `name:`) for unambiguous sender identity matching. `normalizeLegacySenderKey()` handles backward-compatible untyped keys with `warnLegacyToolsBySenderKey()` emitting `DeprecationWarning` for unprefixed entries; untyped keys are now matched as `id` only to prevent cross-field confusion. A compilation step pre-builds typed policy buckets, closing the ambiguous sender-key matching vulnerability.

#### Tighten Elevated allowFrom Sender Matching ([`6817c0ec7`](https://github.com/openclaw/openclaw/commit/6817c0ec7))

Rewrites sender matching in the elevated `allowFrom` feature to support explicit field-typed entries (extracted to `elevated-allowlist-matcher.ts`). `ExplicitElevatedAllowField` and `EXPLICIT_ELEVATED_ALLOW_FIELDS` enumerate the valid match fields (`id`, `from`, `e164`, `name`, `username`, `tag`); `parseExplicitElevatedAllowEntry()` parses `field:value` format entries with strict field validation, rejecting unrecognized field names. The new matcher integrates channel-native identity comparison via per-channel `formatAllowFrom()` hooks, enabling correct matching across channel-specific identity formats.

#### Harden Connect Auth Flow and Exec Policy Diagnostics ([`bbdfba569`](https://github.com/openclaw/openclaw/commit/bbdfba569))

Extracts `resolveConnectAuthDecision()` from `message-handler.ts` into `auth-context.ts`, adding device token rate limiting via `AUTH_RATE_LIMIT_SCOPE_DEVICE_TOKEN` before token verification; `ConnectAuthDecision` captures the auth result, method, and success state as a structured return type. A new `exec-policy.ts` module provides `evaluateSystemRunPolicy()` with `SystemRunPolicyDecision` (a discriminated union for allowed/denied states with structured error reasons including `formatSystemRunAllowlistMissMessage()` for Windows shell wrapper detection); this extraction from `invoke-system-run.ts` improves testability and isolates policy logic.
