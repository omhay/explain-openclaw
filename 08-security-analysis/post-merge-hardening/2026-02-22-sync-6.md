### Key Security Changes

1. **`0bd9f0d4a`** — **Strict Allowlist Enforcement Across Pairing Stores:**

   When an operator sets `dmPolicy: allowlist`, only the statically configured `allowFrom` list should be honored — pairing-store entries (accumulated via `/pair` commands) must not supplement it. A systematic gap existed where `readAllowFromStore()` was called regardless of `dmPolicy`, allowing users who had previously paired to bypass a stricter allowlist policy applied later.

   The fix propagates `dmPolicy` into the core merge helpers and adds a short-circuit guard (`if dmPolicy === 'allowlist' then storeAllowFrom = []`) across every channel integration:

   - `src/channels/allow-from.ts:1` — `mergeAllowFromSources()` now accepts `dmPolicy` and returns empty store entries when `dmPolicy === 'allowlist'`
   - `src/security/dm-policy-shared.ts:5` — `resolveEffectiveAllowFromLists()` excludes `storeAllowFrom` when `dmPolicy === 'allowlist'`
   - `src/plugin-sdk/command-auth.ts:19` — `resolveSenderCommandAuthorization()` adds `params.dmPolicy !== 'allowlist'` guard before reading the store
   - `src/telegram/bot/helpers.ts:20` — `resolveTelegramGroupAllowFromContext()` now receives `dmPolicy` and passes it through to `normalizeAllowFromWithStore()`
   - All 15+ channel extension monitor files updated analogously (Telegram, Discord, Slack, Signal, IRC, iMessage, Mattermost, MS Teams, Matrix, LINE, Google Chat, Feishu, NextCloud Talk, WhatsApp, Bluebubbles)

   **Relevance to audit claims:** Closes a pairing-store bypass of the `allowlist` policy. Addresses **Audit 2 Claim 5** (self-approving / no RBAC — pairing-store bypass) and **Audit 1 Claim 5** (insufficient access control — allowlist integrity).

2. **`71bd15bb4`** — **SSRF CIDR-Table Hardening — Special-Use IPv4 Ranges:**

   The prior SSRF blocklist in `src/infra/net/ssrf.ts` used hand-coded `if` branches (isPrivateIpv4) covering only the six canonical ranges (RFC 1918, loopback, link-local). Several RFC-designated special-use ranges were unguarded, meaning crafted URLs pointing at `198.18.0.1` (IETF benchmark), `192.0.2.1` (TEST-NET-1), `203.0.113.1` (TEST-NET-3), `198.51.100.1` (TEST-NET-2), or multicast/reserved ranges could bypass the SSRF guard.

   The fix replaces isPrivateIpv4 with a CIDR-table approach:

   - `src/infra/net/ssrf.ts:301` — `BLOCKED_IPV4_SPECIAL_USE_CIDRS` constant: 15 CIDR entries (0.0.0.0/8, 10.0.0.0/8, 100.64.0.0/10, 127.0.0.0/8, 169.254.0.0/16, 172.16.0.0/12, 192.0.0.0/24, 192.0.2.0/24, 192.88.99.0/24, 192.168.0.0/16, 198.18.0.0/15, 198.51.100.0/24, 203.0.113.0/24, 224.0.0.0/4, 240.0.0.0/4)
   - `src/infra/net/ssrf.ts:321` — `isBlockedIpv4SpecialUse()` replaces the old isPrivateIpv4 at all call sites
   - `src/infra/net/ssrf.ts:335` — `isPrivateIpAddress()` (exported) updated to call `isBlockedIpv4SpecialUse()`
   - IPv6-mapped (`::ffff:198.18.0.1`) and 6to4-encapsulated variants also blocked via the existing IPv6 extraction path

   **Doc ref line shifts (VERIFIED):** The ~27 new lines shift all downstream exported symbols in `ssrf.ts` — `isPrivateIpAddress` :308→:335, `isBlockedHostnameOrIp` :391→:425, `resolvePinnedHostnameWithPolicy` :429→:496, `resolvePinnedHostname` :391→:544.

   **Relevance to audit claims:** Directly strengthens the DNS rebinding / SSRF defense (**Audit 2 Claim 4**). IPv6-mapped and 6to4 variants also covered. Contributes to Legitimate Gap 1 defense-in-depth (network-layer blocking).

3. **`e23c08b5f`** — **Prototype Pollution Fix in `applyMergePatch`:**

   Adds `isBlockedObjectKey(key)` guard (`src/config/merge-patch.ts:74`) inside `applyMergePatch()` to skip `__proto__`, `prototype`, and `constructor` keys during JSON merge-patch application. Without this guard, a crafted config patch containing `{"__proto__": {"polluted": true}}` could pollute `Object.prototype`. The guard references `BLOCKED_OBJECT_KEYS` from `src/config/prototype-keys.ts:1`.

   **Relevance to audit claims:** Defense-in-depth (no direct claim match). Mitigates CWE-1321 (Prototype Pollution) in the config merge path.

4. **`95dab6e01`** — **Centralized Prototype-Key Guards Across Config System:**

   Extends the `isBlockedObjectKey` guard from `src/config/prototype-keys.ts:1` to all remaining config mutation entry points:

   - `src/config/runtime-overrides.ts:3` — imports and enforces `isBlockedObjectKey` before writing runtime overrides
   - `src/config/config-paths.ts:2` — enforces guard on path key parsing
   - `src/config/includes.ts:18` — enforces guard during deep merge operations
   - `src/config/legacy.shared.ts:15` — enforces guard in legacy config merging

   Ensures no config mutation path can be used as a prototype pollution vector, even through non-standard entry points. This commit (by community contributor @Clawborn) completes the systematic hardening started by `e23c08b5f`.

   **Relevance to audit claims:** Defense-in-depth (no direct claim match). Closes remaining CWE-1321 attack surface in the config system.

5. **`617e38cec`** — **macOS App: Enforce WSS for Non-Loopback Gateway Connections:**

   Enforces `wss://` (WebSocket over TLS) for all non-loopback direct gateway connections in the macOS app. `GatewayDiscoveryHelpers.swift:57` now derives the WebSocket scheme as `self.isLoopbackHost(endpoint.host) ? "ws" : "wss"`, preventing plaintext credential and session-token transmission over unencrypted channels for remote gateway connections.

   **Relevance to audit claims:** Defense-in-depth (no direct claim match). Closes a transport-layer credential exposure vector for remote macOS gateway configurations.

6. **`8942ac04a`** — **macOS App: Fail-Closed on Unauthenticated Discovery Routing:**

   Hardens the macOS app's gateway discovery routing to fail closed when the device is in an unauthenticated state. `GatewayDiscoveryHelpers.swift` (50-line change) and `OnboardingView+Actions.swift`/`NodePairingApprovalPrompter.swift` updated to ensure unauthenticated discovery paths do not route to authenticated gateway state. Prevents a state-confusion condition where an unauthenticated discovery response could influence routing decisions intended for authenticated sessions.

   **Relevance to audit claims:** Defense-in-depth (no direct claim match). Fail-closed security posture strengthened in macOS app discovery flow.

7. **`6ac89757b`** — **Gateway: Control UI Static Path Containment:**

   Adds explicit path containment check before serving any static file from the Control UI: `src/gateway/control-ui.ts:426` calls `isWithinDir(root, filePath)` (from `src/infra/path-safety.ts:8`) after `path.resolve(root, fileRel)`. If the resolved file path escapes the Control UI root directory, the request is rejected before `resolveSafeControlUiFile()` is called. This is a defense layer added ahead of the existing `resolveSafeControlUiFile()` path (which uses `O_NOFOLLOW` and inode verification from `7c500ff62` in sync 5).

   **Relevance to audit claims:** Defense-in-depth (no direct claim match). Strengthens path traversal prevention for the Control UI HTTP server.
