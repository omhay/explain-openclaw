#### Require Explicit Non-Loopback Control-UI Origins ([`223d7dc23`](https://github.com/openclaw/openclaw/commit/223d7dc23d73bdddd7d221d7ddbecafe5d03bfcb))

Breaking change that enforces strict origin validation for non-loopback Control UI deployments. Previously, `checkBrowserOrigin()` accepted any request whose `Origin` header host matched the gateway's own `Host` header — a Host-header-based fallback. This commit gates that fallback behind the explicit `dangerouslyAllowHostHeaderOriginFallback` boolean in `GatewayControlUiConfig`. Non-loopback deployments must now set `gateway.controlUi.allowedOrigins` to a list of trusted origins, or explicitly opt into the legacy Host-header behavior. `resolveGatewayRuntimeConfig()` throws at startup if a non-loopback bind is detected without either allowedOrigins or the fallback flag. `collectGatewayConfigFindings()` adds two new CRITICAL audit checks: missing allowed origins on non-loopback Control UI, and use of the dangerous host-header fallback. `collectEnabledInsecureOrDangerousFlags()` tracks `dangerouslyAllowHostHeaderOriginFallback=true` alongside the existing insecure flags.

#### Harden Browser SSRF Defaults and Migrate Legacy Key ([`5eb72ab76`](https://github.com/openclaw/openclaw/commit/5eb72ab769517e31d96140f1aa66bd1a47a40c2a))

Hardens the browser navigation SSRF policy by defaulting to trusted-network mode and migrating the legacy config key. The old `browser.ssrfPolicy.allowPrivateNetwork` key is preserved for backward compatibility but superseded by `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork`, which must now be explicitly set to allow private-network URLs. A new `assertBrowserNavigationResultAllowed()` guard is added alongside the existing `assertBrowserNavigationAllowed()` — together they check both the pre-navigation URL and the post-redirect final URL, closing a bypass where a redirect from a public URL to a private address could escape the SSRF guard. This directly addresses Audit 2 Claim 4 (DNS rebinding SSRF via web-fetch). A `doctor-legacy-config.ts` migration automatically upgrades old `allowPrivateNetwork` entries.

#### Unify Dangerous Name Matching Handling ([`161d9841d`](https://github.com/openclaw/openclaw/commit/161d9841dc6aae925c7e3d4994f7df1c5030c335))

Extracts per-channel name-matching detection into a new dedicated module `src/security/mutable-allowlist-detectors.ts`. The module exports channel-specific detectors — `isDiscordMutableAllowEntry()`, `isSlackMutableAllowEntry()`, `isGoogleChatMutableAllowEntry()`, `isMSTeamsMutableAllowEntry()`, `isMattermostMutableAllowEntry()`, `isIrcMutableAllowEntry()` — replacing duplicated inline logic spread across channel files and `audit-channel.ts`. Centralizing detection ensures the security audit and the runtime allowlist matching agree on which entry formats trigger the "mutable identity" warning, preventing audit/runtime divergence where a format is accepted at runtime but reported as safe by the auditor.

#### Make allowFrom Id-Only by Default with Dangerous Name Opt-In ([`cfa44ea6b`](https://github.com/openclaw/openclaw/commit/cfa44ea6b4f7ad3210c65a2bc2e18b5754852615))

Critical hardening that flips the default allowFrom matching posture from name+ID to ID-only across Discord, Slack, Google Chat, Mattermost, MS Teams, and IRC. Previously, entries matching mutable identifiers (display names, email addresses, usernames) were accepted by default, allowing an attacker who controls or spoofs a username to match an allowFrom entry belonging to a different user. The commit adds `dangerouslyAllowNameMatching: boolean` as an explicit per-channel opt-in for operators who need legacy name-based matching. This addresses Audit 2 Claims 5-6 (self-approving agent, identity spoofing via allowlist bypass). GHSA-f6h3-846h-2r8w and GHSA-wpph-cjgr-7c39, both related to sender-scoped authorization, are patched by this commit family.

#### Restrict commands.allowFrom to Sender Principals ([`08e2aa44e`](https://github.com/openclaw/openclaw/commit/08e2aa44e78a9c946d97bea62304e6f533b8fa8e))

The `command-auth.ts` allowFrom resolution now enforces that `commands.allowFrom` entries are matched only against the message sender's principal, not against ambient session-level identities. Previously, under certain conditions, an allowFrom check could match a non-sender account ID in the session context, allowing cross-principal command injection. The fix scopes the `allowFrom` lookup to the actual `dock.config.resolveAllowFrom()` result tied to the specific sender, with a fallback to `ownerAllowFrom` only for explicit owner-scoped commands.

#### Harden Exec Wrapper Allowlist Execution Parity ([`a1c4bf07c`](https://github.com/openclaw/openclaw/commit/a1c4bf07c6baad3ef87a0e710fe9aef127b1f606))

Closes a class of exec allowlist bypass via wrapper commands that previously weren't analyzed with the same rigor as direct binary invocations. `exec-approvals-analysis.ts` and `exec-command-resolution.ts` now unwrap `env`, shell dispatch (`sh -c`, `bash -c`), and `--split-string`/`-S` variants before applying allowlist policy, ensuring that `env -S 'banned-cmd args'` and similar wrappers cannot bypass the exec approval gate. `exec-wrapper-resolution.ts` and `exec-approvals-allowlist.ts` are updated for parity. Related to Audit 2 Claim 7 (shell injection via incomplete regex) and several Feb 23 GHSA batch advisories covering exec allowlist bypass patterns.

#### Inherit Dangerous Name-Matching Flag in Mutable Allowlist Scan ([`22467902e`](https://github.com/openclaw/openclaw/commit/22467902ea54c650b1bb30cef54d83b6cc2ff017))

Previously, the doctor scan ran the mutable-entry check without the channel's name-matching flag, causing false positives (entries reported as dangerous when the operator had explicitly opted in) and potentially false negatives in edge cases. #### Harden ACP Permission Tool-Name Validation ([`63dcd28ae`](https://github.com/openclaw/openclaw/commit/63dcd28ae0be2de1c75af09cc81841cebeec068f))

`src/acp/client.ts` adds explicit tool-name validation to the ACP permission grant path. Before this fix, a malformed or attacker-crafted tool name containing special characters could slip through the permission check and match a broader tool policy than intended, enabling a tool-name spoofing escalation within ACP permission grants. The fix validates tool names against an allowable-format pattern before applying permission decisions.

#### Harden Account-Key Handling Against Prototype Pollution ([`f97c0922e`](https://github.com/openclaw/openclaw/commit/f97c0922e1d8da6aca4c113408d7b23234cff983))

Introduces `isBlockedObjectKey()` (from `src/infra/prototype-keys.ts`) as a guard across 20 production files in the account/config routing layer. The check blocks reserved JavaScript prototype keys — `__proto__`, `constructor`, and `prototype` — from being used as account IDs, channel keys, or config object keys. Without this guard, an attacker controlling an account ID string could pollute Object.prototype properties in the routing tables that store per-account channel configs and allowlists, potentially overriding default values for all accounts. The fix is applied at `resolveAccountTarget()` in `src/auto-reply/reply/commands-allowlist.ts` and at equivalent routing points across auto-reply, chunk processing, and block-streaming paths.

#### Harden ACP Permission Auto-Approval Policy ([`12cc75433`](https://github.com/openclaw/openclaw/commit/12cc754332f9a7c92e158ce7644aa22df79c0904))

`src/acp/client.ts` tightens the conditions under which ACP permission requests are auto-approved. Previously, the auto-approval logic had edge cases where a narrowly scoped permission request could be auto-approved even when the operator's policy should have required explicit confirmation. The fix adds additional checks in the auto-approval gate, aligning with Audit 2 Claim 5 (self-approving agent without RBAC). Documentation in `docs/cli/acp.md` is updated to reflect the stricter policy.

#### Harden Catchall Hint Mapping and Array Fallback ([`13478cc79`](https://github.com/openclaw/openclaw/commit/13478cc79a8070b99613e48183675a3552eacaaa))

`src/config/schema.hints.ts` and `src/config/redact-snapshot.ts` are hardened against prototype-adjacent injection via the config hint mapping path. The catchall hint resolution previously used object-property access patterns that could be manipulated via specially crafted config keys. The refactor ensures array fallback and hint lookup use safe property access patterns, preventing hint-map injection from influencing config schema metadata displayed to users or operators.

#### Disable Developer Role for DashScope-Compatible Endpoints ([`30c622554`](https://github.com/openclaw/openclaw/commit/30c622554f413131baf570a2dd3d955a25f71d6e))

DashScope-compatible provider endpoints no longer receive the `developer` role, which grants elevated tool-use privileges not appropriate for third-party provider connections. Previously, all providers using the DashScope-compatible API format inherited the same role assignment as first-party providers, creating a privilege escalation path where a malicious or compromised DashScope-compatible endpoint could influence tool approval flows. The fix restricts the `developer` role to explicitly trusted providers.
